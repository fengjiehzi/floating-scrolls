# AI 小说工作流

使用项目已有资料和脚本推进创作。不要复制规则、建立平行台账或把未审草稿写成既成事实。

## 定位项目

从当前目录向上查找同时包含以下路径的最近目录，将其作为项目根目录：

- `00_项目管理/项目状态.md`
- `01_小说圣经/小说圣经.md`
- `02_故事规划/故事总纲.md`
- `02_故事规划/连续性台账.md`
- `scripts/new-chapter.ps1`

找不到时，说明当前目录不是已初始化的小说项目，并询问项目路径。不要自行新建另一套工作流。

## 每次必读

1. 读取根目录 `AGENTS.md`。
2. 读取 `00_项目管理/项目状态.md`。
3. 根据操作按下表加载资料，不一次性读取所有章节。

| 操作 | 额外读取 |
|---|---|
| 立项 / 规划 | `小说圣经.md`、`故事总纲.md` |
| 建章 / 写章 | `小说圣经.md`、总纲相关卷与章节、`连续性台账.md`、当前章 `任务卡.md`；必要时上一章结尾 |
| 审章 / 退修 | 上述资料、当前章 `正文.md`、`05_质量控制/检查清单.md` |
| 定稿 | 当前章 `任务卡.md`、`正文.md`、`审稿.md`、`连续性台账.md`、总纲章节路线图、项目状态 |

## 选择动作

优先执行用户明确指定的动作。用户只说“继续”或“下一步”时，根据项目状态选择第一个未通过的闸门：

1. 小说圣经或结局方向未确定：补充立项资料。
2. 当前卷或当前章在总纲中未规划：补充规划。
3. 当前章目录不存在：创建章节。
4. 任务卡未完成：填写任务卡。
5. 正文为空：写章。
6. 审稿状态为“待审”：审章。
7. 审稿状态为“退修”：按必须修改项局部修改后复审。
8. 审稿状态为“通过”且台账未同步：定稿回写。
9. 全部完成：创建下一章；缺少必要的章节编号或故事方向时只问一个最关键问题。

## 执行动作

### 立项与规划

- 从用户已提供的题材、人物、冲突和结局方向填写现有模板。
- 不替用户决定核心题材、价值答案、主角终局或内容边界；信息不足时集中询问最少的关键问题。
- 先确定结局与全书因果链，再拆分卷和临近章节。
- 远期章节保持简洁，避免伪精确规划。

### 创建章节

使用现有脚本：

```powershell
.\scripts\new-chapter.ps1 -Number <章节号> -Title "<章节标题>"
```

优先从总纲章节路线图取得编号和标题。目录已存在时使用现有目录，不创建重复章节。

### 填写任务卡

- 将总纲中的章节目标转化为一个可判定完成与否的核心任务。
- 明确章初状态、章末状态、场景因果链、知识边界、禁止事项和章末钩子。
- 任务卡不得改变小说圣经、既成事实或总纲主线。

### 写章

- 严格执行 `04_提示词/写章提示词.md`。
- 直接编辑当前章 `正文.md`。
- 发现资料冲突时停止扩写并标记 `【待确认】`，不要自行圆设定。
- 写完后依据检查清单自检，只修正文，不更新台账。

### 审章与退修

- 严格执行 `04_提示词/审章提示词.md`。
- 将带具体证据的问题写入当前章 `审稿.md`。
- 任何硬性一致性问题存在时结论必须为“退修”。
- 退修只改能解决根因的最小范围；修改后重新审查受影响项。

### 定稿

仅当 `审稿.md` 明确为“通过”时执行：

1. 将正文已确认的时间、位置、人物状态、知识、关系、物品、伏笔和未解决问题写入 `连续性台账.md`。
2. 将故事总纲章节路线图中的该章状态改为 `定稿`。
3. 更新 `项目状态.md` 的最近定稿章、当前章节、下一步动作和工作摘要。
4. 若正文改变核心规则，先记录为冲突并要求确认；不要在定稿阶段静默改小说圣经。

## 完成检查

修改模板或脚本后运行：

```powershell
.\scripts\test-workflow.ps1
```

普通内容修改无需运行脚本测试，但必须检查 Markdown 表格、章节编号和状态字段是否一致。

最后只汇报：完成了什么、修改了哪些文件、当前闸门、下一步唯一动作。
