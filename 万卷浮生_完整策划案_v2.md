## 第一章：产品定位与核心哲学

### 1.1 一句话产品定义

> **万卷浮生——让玩家进入小说，养成角色，改写命运。**

20字以内的核心定义：
> **「进小说，养成角，改命运」**

### 1.2 产品核心身份

**身份：游戏（选A）**

万卷浮生的核心是**小说世界叙事养成与对战**。AI 造卡、小说解析和世界书生成是降低内容生产门槛的手段，最终目标是：

```
进入小说世界 → 养成角色 → 改写剧情 → 验证强弱 → 分享与传播
```

**为什么不是"工具"或"社区"**：
- 工具型产品用户用完即走，没有持续动力
- 社区型产品需要先有大量内容，MVP阶段难以实现
- **游戏是唯一能让用户愿意每天回来、持续投入、并主动分享的形态**

### 1.3 差异化壁垒——"体验链路"，不是单点算法

万卷浮生的真正壁垒**不是**单点AI算法，而是**完整体验链路**：

| 单点能力 | 容易被复制吗 | 我们的优势 |
|----------|--------------|------------|
| AI 提取角色 | ✅ 容易被复制 | 大厂接入更好模型即可 |
| 战力量化 | ✅ 容易被复制 | 算法是公开的 |
| PVP 战斗引擎 | ✅ 容易被复制 | 战斗系统是标准能力 |
| **完整体验链路** | ❌ 难以复制 | 需要产品哲学 + 多模块协同 |
| **用户创造的新剧情被解析和反馈** | ❌ 难以复制 | 需要长期数据积累 |
| **玩家对剧情走向的归属感** | ❌ 难以复制 | 是情感和社交资产 |

**关键判断**：
> 单点 AI 算法容易被追上，但如果用户能持续创造新剧情、养成角色、验证强弱，并且看到系统对自己创造剧情的解析和反馈，**这套体验更难复制**。

### 1.4 古籍活化的战略定位

**古籍活化是重要加分项，不是产品限制。**

| 维度 | 立场 |
|------|------|
| 当前选择 | 优先使用古籍和名著 |
| 比赛契合 | 古籍活化能展示文化价值，契合大赛 |
| 长期方向 | **不只做名著，以小说为主** |
| 战略意义 | 证明经典文本可以被重新进入、重新演绎、重新游玩 |

**关键判断**：
> 古籍活化要服务产品表达，但它不能限制产品未来只做名著。MVP 阶段做古籍是因为它适合比赛赛道，也更容易展示文化价值和公共认知。

### 1.5 产品的"四不像"反而是优势

| 常见品类 | 万卷浮生的位置 | 借鉴之处 |
|----------|---------------|----------|
| 文字冒险游戏 | ✅ 核心 | 剧情分支、选择影响、养成感 |
| 卡牌对战游戏 | ✅ 核心 | 角色卡、技能、数值平衡 |
| AI 工具 | ✅ 工具 | 小说解析、角色生成 |
| 内容社区 | ✅ 平台 | UGC 剧情、战报分享 |
| 桌游/跑团 | ✅ 借鉴 | 公开规则、骰子裁决、戏剧性 |

**产品哲学**：
> 万卷浮生**不模仿任何单一产品**，而是把"小说沉浸 + 角色养成 + 自动推演 + 叙事解释"四个独立领域的能力，**整合成一条独特体验链路**。

## 第二章：目标用户与需求分析

### 2.1 用户结构与优先级

| 优先级 | 用户类型 | 占比 | 早期打动力 | 系统要求 |
|--------|---------|------|-----------|----------|
| **P0 核心** | 小说/动漫粉丝 | 40% | 强（对角色有感情） | 角色还原度、剧情代入感 |
| **P1 次核心** | 战力讨论爱好者 | 30% | 中（喜欢讨论强弱） | 战斗可解释、结果可复盘 |
| **P2 潜力** | 内容创作者 | 20% | 强（需要素材） | 可分享的战报、可二创的剧情 |
| **P3 边缘** | 卡牌对战玩家 | 10% | 弱（对平衡要求高） | 平衡性、深度系统 |

**关键判断**：
- **早期最能被打动的是对角色和剧情有感情的人**（小说/动漫粉丝）
- 卡牌对战玩家对平衡和系统深度要求更高，**不应作为最早核心用户**
- 战力讨论爱好者是天然传播者，会主动把"孙悟空vs迪迦"结果分享出去

### 2.2 种子用户来源（开源项目版）

| 渠道 | 类型 | 获取方式 | 吸引点 |
|------|------|----------|--------|
| **GitHub** | P0 | README、Demo、技术文章 | 可 fork、二创、技术展示 |
| **开源社区** | P0 | V2EX、掘金、CSDN | AI + 游戏化完整链路 |
| **AI 应用社区** | P0 | Prompt 调优社区、AI 工具圈 | 小说解析、角色生成 prompt |
| **开发者社群** | P1 | Discord、QQ 群 | 可部署、本地运行 |
| **B站/知乎** | P2 | 演示视频、科普文章 | 跨作品战力讨论话题 |

**核心策略**：
> 早期不做大众推广，**先把项目做成可部署、可二创、可试玩的开源项目**。通过 README、Demo、技术文章和开源传播吸引第一批种子用户。

### 2.3 用户动机的深度分析

#### 2.3.1 留下来——什么是"第二天还来"的真正动力？

| 表面动机 | 深层动机 | 系统设计 |
|---------|---------|----------|
| 数值涨了 | 自己的角色正在成长 | 养成进度可视化 |
| 解锁新技能 | 系统理解我的角色 | 解锁提示带叙事 |
| 打完一场 | 验证昨天的选择是否对 | 战斗结果关联选择 |
| **看到剧情分支** | **我创造的新剧情正在被理解** | **世界书实时反映玩家行为** |

**用户留下来的真正关键**：
> 第二天回来，不只是为了看数值涨了，而是想知道自己上一轮选择会如何影响剧情、人物关系、世界状态和后续命运。系统要让用户感到：我的角色正在成长，我创造的新剧情正在被理解，后面还有新的分支、新的关系、新的战斗和新的结局。

#### 2.3.2 分享出去——什么会让人主动传播？

| 分享动机 | 触发场景 | 传播效果 |
|---------|---------|----------|
| **我改写的剧情走向** | 玩家在分支中选到离谱/精彩/意外剧情 | 长尾传播，引发讨论 |
| **谁更强的讨论** | 跨作品对战结果意外 | 争议驱动转发 |
| AI 造卡 | "AI居然把XX做出来了" | 单次惊叹 |
| 邀请对战 | "快来和我打一场" | 单次邀请 |

**最强分享动机**：
> 用户会分享自己改写出的离谱、精彩或意外剧情，也会分享角色对战结果，**引发朋友讨论这个剧情是否合理、这个角色是否应该更强、系统给出的胜负解释是否说得通**。

### 2.4 用户旅程地图

```
认知阶段
   ↓
  发现：GitHub README / B站视频 / 朋友推荐
   ↓
  试用：下载 Demo / 在线体验 / Fork 仓库
   ↓
  首次体验：上传小说 → 看到 AI 解析 → 第一个角色被造出
   ↓
留存阶段
   ↓
  第二天：养成角色 → 看到剧情分支
   ↓
  第一周：解锁形态 → 第一次 PVP
   ↓
  第一个月：创造自己的剧情改写
   ↓
传播阶段
   ↓
  分享：把"离谱的剧情"截图发到群里
   ↓
  邀请：让朋友上传自己的本命小说
   ↓
  共创：参与角色设计/技能设计讨论
   ↓
忠实用户
   ↓
  长期：拥有多个养成角色，持续参与新剧情
```

## 第三章：核心功能模块设计

### 3.1 系统模块总览

万卷浮生由 **7大核心模块** 组成，形成完整体验链路：

```
┌─────────────────────────────────────────────────────┐
│  1. 小说解析引擎（Novel Parser）                    │
│     - 公版小说解析                                  │
│     - 世界书生成                                    │
│     - 角色/事件/关系抽取                            │
└─────────────────┬───────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────────────┐
│  2. AI 角色生成器（Character Forge）                │
│     - 角色卡生成（属性/技能/形态）                  │
│     - 玩家编辑修正                                  │
│     - 数值校验                                      │
└─────────────────┬───────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────────────┐
│  3. 世界书系统（Worldbook）                          │
│     - 场景/势力/规则管理                             │
│     - 用户创造剧情的记录                            │
│     - 实时反映玩家选择                              │
└─────────────────┬───────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────────────┐
│  4. 角色养成系统（Character Growth）                 │
│     - 等级/经验/技能升级                            │
│     - 形态解锁                                      │
│     - 剧情选择影响成长方向                          │
└─────────────────┬───────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────────────┐
│  5. 战前策略系统（Pre-battle Strategy）              │
│     - 形态/技能/战术倾向选择                        │
│     - 装备/法宝配置                                  │
│     - AI 推演与叙事解释                              │
└─────────────────┬───────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────────────┐
│  6. 战斗推演引擎（Battle Engine）                    │
│     - 12 回合自动战斗                                │
│     - 等级/次元压制                                  │
│     - 战斗日志与高亮战报                            │
└─────────────────┬───────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────────────┐
│  7. 叙事解释系统（Narration Engine）                 │
│     - 胜负解释（"为什么赢/为什么输"）               │
│     - 关键回合戏剧化                                │
│     - 可分享战报生成                                │
└─────────────────────────────────────────────────────┘
```

### 3.2 模块详解

#### 模块 1：小说解析引擎

**核心能力**：

| 能力 | 实现方式 | 准确率预期 |
|------|----------|-----------|
| 文本分章节 | 按章节标记切分 | 95%+ |
| 角色识别 | NLP 提取 + 大模型校验 | 70-80% |
| 关系抽取 | 大模型 + 规则 | 60-70% |
| 战力描写识别 | 规则 + 关键词 | 50-60% |
| 世界观总结 | 大模型生成 | 80-90% |

**输入**：公版小说（txt / epub / 粘贴）
**输出**：世界书 JSON + 角色卡候选列表

**MVP 限制**：
- 优先支持四大名著 + 经典小说（公版）
- 长文本分块处理
- 解析结果需要玩家确认

#### 模块 2：AI 角色生成器

**核心能力**：

```
AI 生成初稿（属性/技能/形态）
        ↓
玩家编辑修正（AI 初稿 + 玩家编辑）
        ↓
系统校验（数值范围/技能冲突/规则一致）
        ↓
正式入库（可参与对战/养成）
```

**设计哲学**：
> AI 造卡失败时，采用"AI 初稿 + 玩家编辑修正"的方式。AI 负责生成基础角色、技能和数值，玩家可以修正不准的地方，系统再做基础校验和标记。这样既承认 AI 可能出错，也让玩家参与到角色共创里。

**数值校验规则**：
- 等级 S/A/B/C 对应数值范围
- 技能不能超过 5 个
- 形态不超过 3 个
- HP/MP 在合理区间

#### 模块 3：世界书系统

**核心创新**——世界书不只是"设定集"，而是**会被玩家行为改变的活的世界**：

```json
{
  "worldbook_id": "xiyouji_v1",
  "title": "西游记",
  "era": "唐代",
  "rules": ["仙魔共存", "佛法无边", "因果轮回"],
  "factions": [
    {"name": "天庭", "stance": "秩序", "key_members": [...]},
    {"name": "花果山", "stance": "自由", "key_members": [...]}
  ],
  "key_events": [
    {"event": "大闹天宫", "year": "贞观13年", "impact": "天庭权威受损"}
  ],
  "user_modifications": [
    {
      "player_id": "u_123",
      "modification": "孙悟空选择保护唐僧而非自立为王",
      "consequence": "齐天大圣形态未解锁，斗战胜佛提前",
      "timestamp": "2026-06-20T10:00:00Z"
    }
  ]
}
```

**关键判断**：
> 用户留下来的关键是养成进度，以及用户对自己创造的新剧情解析。世界书系统要**实时反映玩家的选择**，让用户感到"我的故事正在被记录"。

#### 模块 4：角色养成系统

**核心机制**：

| 养成维度 | 玩家操作 | 系统反馈 |
|----------|----------|----------|
| 等级 | 战斗胜利获得经验 | 自动升级，解锁新内容 |
| 经验 | 参与剧情、使用角色 | 累积到下一级 |
| 技能升级 | 选择技能投入资源 | 数值提升，叙事更新 |
| 形态解锁 | 推进剧情到关键节点 | 形态可用，战斗前可切换 |
| 剧情选择 | 在节点图中做选择 | 影响属性成长方向 |

**养成路线设计**：
- **攻击路线**：偏力量、特殊能力，解锁"大闹天宫"类形态
- **防御路线**：偏防御、智力，解锁"斗战胜佛"类形态
- **均衡路线**：全属性+少量加成，解锁多个形态

**关键设计原则**：
> 同一角色可多次养成（不同存档），探索不同路线。每次养成都是新故事。

#### 模块 5：战前策略系统

**玩家在战前的选择**：

```
出战角色选择
   ↓
形态选择（基础/进阶/终极）
   ↓
技能配置（5选3或类似）
   ↓
战术倾向（进攻/防守/平衡）
   ↓
装备/法宝（可选）
   ↓
确认 → 进入战斗推演
```

**关键设计**：
> 战前策略让玩家有参与感，但战中由系统自动推演。这样既有参与感，又不需要做复杂的即时操作，也更适合保留 AI 推演和叙事解释的特色。

#### 模块 6：战斗推演引擎

**核心实现**（基于现有 battle-engine.js）：

```
回合循环（最多12回合）：
   1. 按速度决定先手
   2. AI 选技能（基于血量/MP/技能列表）
   3. 伤害计算
      base = 属性 × 1.2
      damage = base × skillMult × critMult × defReduction × gradeMult × dimensionMult
   4. 判定胜负
   5. 清理 buff
   6. 进入下一回合
```

**关键公式**（已实现）：

| 公式 | 计算方式 |
|------|----------|
| 暴击率 | `min(0.35, (speed + intelligence) / 400)` |
| 防御减免 | `1 - (defense / (defense + 200))` |
| 等级压制 | S>A>B>C，差1级 1.5x，差2级 2.0x |
| 次元压制 | 神话打凡人 3.0x，凡人打神话 0.5x |
| 秒杀判定 | 伤害 > 目标 HP × 60% 触发⚡秒杀 |

**平衡 vs 真实的处理**：
> 这个游戏不需要太追求传统意义上的平衡性。更重要的是有一套公开、一致的基准规则。在同一基准下，强弱差异应该自然体现出来，这样才符合跨作品战力讨论的核心乐趣。只要规则透明、标准一致，强角色就应该强，弱角色也可以通过特定条件、战术或剧情解释获得少量机会，但不需要强行五五开。

#### 模块 7：叙事解释系统

**这是产品的灵魂模块**——战斗不只是结算胜负，而是要把胜负过程讲得精彩、合理、有戏剧性。

**叙事生成的三个层次**：

1. **回合级**："孙悟空施展筋斗云突袭，瞬间逼近迪迦，造成152点伤害"
2. **关键节点级**："第三回合，迪迦切换强力型，防御骤然提升，孙悟空的连击效果骤降"
3. **胜负级**："经过5回合激战，孙悟空以微弱优势险胜。这场对决的胜负关键在于第二回合孙悟空抢到了先手，打出了关键暴击……"

**秒杀局叙事**：
> 这一击挟神话级威能，凡人之躯难以承受。关羽虽勇，终究无法跨越次元的鸿沟。
> ⚡ 仅1回合，孙悟空便以碾压之势秒杀关羽！

**核心哲学**：
> 核心爽点是叙事解释。数值只是底层结果，真正打动用户的是系统如何解释角色为什么赢、为什么输、关键回合为什么发生。让用户觉得"这个结果说得通，甚至值得转发"。

### 3.3 战斗观赏性升级

**现状评价**：血条 + 文字日志不够，适合 Demo 验证，但不够支撑长期传播。

**理想中的战斗画面**：

| 元素 | 实现方式 | 优先级 |
|------|----------|--------|
| 角色立绘 | SVG/AI 生图（参考现有 14 个 SVG） | P0 |

## 第四章：产品体验链路设计

### 4.1 完整体验链路图

这是产品的**核心差异化和壁垒**：

```
┌────────────────────────────────────────────────────────────┐
│                                                            │
│   [进入小说]                                                │
│       ↓                                                    │

│       ↓                                                    │
│   [选择本命角色]                                            │
│       ↓                                                    │
│   AI 生成角色卡 + 玩家微调                                 │
│       ↓                                                    │
│   [进入世界]                                                │
│       ↓                                                    │
│   养成角色（剧情选择 + 经验积累）                            │
│       ↓                                                    │
│   [关键剧情节点]                                            │
│       ↓                                                    │
│   世界书更新（用户的改写被记录）                            │
│       ↓                                                    │
│   [战前策略]                                                │
│       ↓                                                    │
│   形态/技能/战术选择                                        │
│       ↓                                                    │
│   [战斗推演]                                                │
│       ↓                                                    │
│   自动战斗 + 战斗日志                                       │
│       ↓                                                    │
│   [叙事解释]                                                │
│       ↓                                                    │

│       ↓                                                    │

│       ↓                                                    │
│   一键分享到社交平台                                        │
│       ↓                                                    │
│   [回到养成]                                                │
│       ↓                                                    │
│   剧情继续发展、新角色解锁、新分支开放                      │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 4.2 链路设计原则

#### 原则 1：每个环节都要"服务叙事"
- AI 解析 → 帮用户理解小说的世界
- 角色生成 → 让用户感觉"我的角色"被创造
- 养成 → 让用户感觉"我的角色在成长"
- 战斗 → 让用户感觉"我的角色的强弱被验证"
- 分享 → 让用户感觉"我的故事值得被传播"

#### 原则 2：用户的创造要被系统"理解"和"反馈"
- 用户在剧情分支中的选择 → 影响世界书
- 用户对角色的微调 → 影响后续推演
- 用户的战斗结果 → 关联回剧情

**核心承诺**：
> 你创造的新剧情正在被理解，后面还有新的分支、新的关系、新的战斗和新的结局。

#### 原则 3：保留"轻量化入口"
- 用户不必从零开始
- 预置角色库（14 个 + 持续扩展）让用户可以快速体验
- 选择"用预置角色"vs"上传小说 AI 解析"两条路径

### 4.3 MVP 阶段的核心闭环

```
公版小说 → AI 解析 → 角色卡列表 → 选择角色 → 战前策略 → 战斗推演 → 叙事解释 → 分享战报
                                              ↑
                                              └── 回到角色库（换角色再来）
```

**MVP 范围说明**：
- 第一版只跑通这条最小闭环
- 养成系统是简化版（只支持等级和形态）
- 剧情选择是预置的（不做动态生成）
- 但**叙事解释**必须做到位

## 第五章：技术实现路径

### 5.1 技术栈演进路线

| 阶段 | 时间 | 技术栈 | 目的 |

### 5.2 桌面化（Godot 4）的理由

| 个人书库管理 | ✅ 完整 | ❌ 受限 |
| 大量文本数据缓存 | ✅ 充足 | ❌ 受限 |
| 本地模型调用 | ✅ 方便 | ❌ 困难 |
| 离线运行 | ✅ 完全 | ❌ 依赖网络 |
| 存档管理 | ✅ 系统级 | ⚠️ 受限 |
| 动画演出 | ✅ 完整 | ⚠️ 受限 |
| 音效反馈 | ✅ 完整 | ⚠️ 受限 |

**未来重构方向**：
> 以 Godot 4 桌面应用为唯一主形态，重新设计本地书库、AI 服务配置、世界书管理、角色养成、战斗推演、存档、导出分享和开源部署能力。

### 5.4 AI 接入策略

**MVP 阶段就接入真正的大模型**：

| 用途 | 模型 | 降级方案 |
|------|------|----------|
| 小说解析 | GPT-4 / Claude / GLM-4 | 规则+NLP（保底） |
| 角色生成 | GPT-4 / Claude | 规则（保底） |
| 战斗叙事 | GPT-4 / Claude | 模板（保底） |
| 世界书生成 | GPT-4 / Claude | 规则（保底） |

**成本控制策略**：
- 长文本分块处理
- 结果缓存（相同小说复用）
- 限额机制（每日免费次数）
- 队列和并发控制
- 玩家可自带 API Key

**关键判断**：
> 大模型主要用于世界书解析、角色总结、技能生成、剧情解释和战斗叙事增强。纯规则/NLP 只能作为保底，不应该承担最终质量。

### 5.5 性能瓶颈与优化

**最先成为瓶颈的是 AI 提取/生成**：

| 瓶颈点 | 触发场景 | 优化策略 |
|--------|----------|----------|
| 长文本解析 | 用户上传百万字小说 | 分块、并发、缓存 |
| 大模型调用 | 高峰期排队 | 队列、限流、异步 |
| 批量生成 | 一次性造 10 个角色 | 增量、缓存 |
| 成本失控 | 大量用户同时使用 | 限额、降级策略 |

**次要瓶颈**：
- 数据库写入（采用 WAL 模式已优化）
- 实时连接（WebSocket 状态管理）
- 前端展示（CSS 动画、SVG 优化）

**未来桌面版的优势**：
- 本地模型调用（零网络成本）
- 本地缓存（不依赖云端）
- 离线运行（不依赖网络）

### 5.6 数据安全与版权

| 风险 | 当前策略 | 未来桌面版 |
|------|----------|------------|
| 版权问题 | 优先使用公版/授权；用户上传默认私有 | 全部本地存储，不上传云端 |
| 隐私问题 | 数据绑定用户；最小化云端存储 | 默认本地，零云端传输 |
| 内容安全 | 基础限制 + 举报/删除机制 | 不涉及公开传播 |
| 公开分享 | 需用户主动确认 + 基础审核 | 用户自主选择分享方式 |

**关键判断**：
> 公开展示优先使用公版小说、古籍或授权内容；用户上传内容默认私有，不进入公共内容池。未来桌面版应优先本地保存，减少不必要的云端传输。

### 5.7 开源项目的定位

**定位：可部署产品**

- 可以基于它做自己的小说互动项目

| **重构风险** | 桌面化难度超预期 | 中 | 高 | P1 | 分阶段重构，可回退 |
| **社区风险** | 贡献者少，难以形成生态 | 中 | 低 | P2 | 保持低门槛，文档完善 |
| **比赛风险** | 比赛未获奖，影响信心 | 中 | 低 | P2 | 价值在于产品本身 |

### 8.2 核心风险详解

#### 风险 1：AI 造卡质量（最大风险）

**问题描述**：
- 纯规则/NLP 准确率只有 40-60%
- 玩家对"本命角色被识别错"极度敏感
- 大模型可能产生不存在的技能或数值

**应对策略**：

```
策略 A：AI 初稿 + 玩家修正（已确定）
  - AI 负责生成基础内容
  - 玩家可以修改描述、技能、数值
  - 系统做基础校验

策略 B：多模型集成
  - 不同模型各取所长
  - 投票/交叉验证

策略 C：玩家共建角色库
  - 优秀修正可分享到公共库
  - 形成 UGC 飞轮
```

**关键判断**：
> 早期接受 AI 不完美，但通过玩家修正机制让用户参与到角色共创中。这反而是产品的特色，不是缺陷。

#### 风险 2：战斗体验（核心风险）

**问题描述**：
- 纯数值战斗可能无聊
- 跨作品对战天然不平衡
- 没有操作感，玩家可能感觉"与我无关"

**应对策略**：

| 策略 | 说明 | 优先级 |
|------|------|--------|
| 强化叙事解释 | 让每场战斗都有故事 | P0 |
| 战前策略选择 | 让玩家有参与感 | P0 |
| 可分享战报 | 让战斗结果可传播 | P1 |
| 视觉特效升级 | 让战斗更"好看" | P1 |
| 关键回合慢镜头 | 增强戏剧性 | P2 |

**关键判断**：
> 战斗的核心是叙事解释，不是数值计算。系统要解释"为什么赢、为什么输、关键回合为什么发生"，让用户觉得"这个结果说得通，甚至值得转发"。

#### 风险 3：传播乏力

**问题描述**：
- 项目没人知道
- 早期用户极少
- 难以形成口碑

**应对策略**：

| 阶段 | 策略 |
|------|------|
| 比赛期 | 比赛平台曝光 + B站/知乎内容 |
| 比赛后 | GitHub 持续维护 + 开发者社区 |
| 桌面化后 | 多平台发布 + 玩家社区 |

**开源项目的特殊优势**：
- 不依赖单一渠道
- 技术社区自带传播
- fork 和二创带来自然增长

#### 风险 4：桌面重构超预期

**问题描述**：
- Godot 4 学习成本
- 从 Web 到桌面的全面重构
- 工作量可能是 Web 版的 2-3 倍

**应对策略**：

```
分阶段重构：
  阶段 1：Godot 4 原型（验证可行性） 2026-Q4
  阶段 2：核心功能迁移              2027-Q1
  阶段 3：完整产品实现                2027-Q2
  阶段 4：优化与发布                  2027-Q3

可回退设计：
  - Web 版继续维护
  - 桌面版不达预期则保留双版本
  - 保持核心业务逻辑独立
```

### 8.3 非技术风险

| 风险 | 应对 |
|------|------|
| 个人精力不足 | 设定最小可行版本，避免完美主义 |
| 比赛后失去动力 | 强调"做给玩家用，不是做给评委看" |
| 法律问题 | 公版优先，开源协议清晰 |
| 长期维护 | 保持简单，依赖成熟开源库 |

### 8.4 关键判断

> **对于开源项目，最大的风险不是"没人用"，而是"做得太重"。** 保持小而美、持续迭代、文档完善，是开源项目长期健康的根本。

## 附录：核心决策记录

### 决策 1：产品身份

| 选项 | 选择 | 理由 |
|------|------|------|
| A. 游戏 | ✅ | 用户愿意每天回来、持续投入、主动分享 |
| B. 工具 | ❌ | 用完即走，没有持续动力 |
| C. 社区 | ❌ | MVP 阶段难以实现 |

### 决策 2：差异化壁垒

| 选项 | 选择 | 理由 |
|------|------|------|
| 算法 | ❌ | 容易被追上 |
| 内容 | ❌ | 可以被复制 |
| 体验 | ✅ | 完整体验链路难以复制 |
| 玩家共创 | ✅ | 长期数据积累和情感资产 |

### 决策 3：古籍活化的位置

| 选项 | 选择 | 理由 |
|------|------|------|
| 核心 | ❌ | 不应限制产品方向 |
| 重要加分项 | ✅ | 适合比赛、利于传播 |
| 营销噱头 | ❌ | 浪费了文化价值 |

### 决策 4：战斗体验定位

| 选项 | 选择 | 理由 |
|------|------|------|
| 纯观赏 | ⚠️ | 太被动，但保留观赏性 |
| 战前策略 | ✅ | 玩家有参与感，AI 推演有特色 |
| 战中操作 | ❌ | 复杂，与 AI 推演冲突 |

### 决策 5：AI 造卡失败处理

| 选项 | 选择 | 理由 |
|------|------|------|
| 完全 AI | ❌ | 玩家会失望 |
| AI + 玩家微调 | ✅ | 修正 AI 错误，玩家共创 |
| 完全自定义 | ❌ | 失去战力系统可信度 |

### 决策 6：技术栈演进

| 阶段 | 技术栈 | 决策 |
|------|--------|------|
| MVP | Web (Node.js) | 比赛 Demo、开源传播 |
| 桌面版 | Godot 4 | 性能、本地化、可玩性 |
| 长期 | 仅桌面 | 避免双端维护 |

### 决策 7：开源定位

| 选项 | 选择 | 理由 |
|------|------|------|
| 技术 Demo | ❌ | 不够完整 |
| 可部署产品 | ✅ | fork 即用，二创友好 |
| 社区项目 | ⚠️ | 自然形成，不强求 |

### 决策 8：核心用户优先级

| 优先级 | 用户类型 | 理由 |
|--------|---------|------|
| P0 | 小说/动漫粉丝 | 早期最容易被打动 |
| P1 | 战力讨论爱好者 | 天然传播者 |
| P2 | 内容创作者 | 后期重要 |
| P3 | 卡牌对战玩家 | 平衡要求高，不应早期 |

## 📜 结语

万卷浮生的核心不是"AI 造卡 + PVP 对战"，而是：

> **让玩家进入小说，养成角色，改写命运。**

我们相信：
- 用户的创造应该被系统理解和反馈
- 战斗的胜负应该被精彩地解释
- 跨作品的角色应该可以真正"打一场"
- 经典文本应该被重新进入、重新演绎、重新游玩

通过 **完整的体验链路**（小说解析 → 世界书 → 角色养成 → 战前策略 → 战斗推演 → 叙事解释 → 分享），万卷浮生正在构建一个难以复制的独特产品。

**短期**：完成 Web MVP，参加 TRAE 大赛，开源传播
**长期**：重构为 Godot 4 桌面应用，做成本地小说世界工作台

**这不只是一个游戏，而是小说与玩家的新型互动方式。**
