# 0002: 采用事件总线 + Zustand状态同步的桥接架构

## 状态

已采纳

## 上下文

「万卷浮生」项目采用混合架构：核心业务逻辑（战斗引擎、剧情引擎）保持原生JS实现，UI层和组件层使用React重构。需要定义明确的通信契约，确保原生JS引擎能够与React组件正确交互。

## 决策

采用 **事件总线 + Zustand状态同步** 的双层通信架构，实现原生JS引擎与React组件的解耦通信。

## 备选方案

### 方案 A：仅使用事件总线
- **优点**：实现简单、完全解耦
- **缺点**：无法处理跨组件共享状态、状态变更难以追踪

### 方案 B：仅使用 Zustand
- **优点**：状态管理统一、易于调试
- **缺点**：不适合实时事件、引擎需要直接调用React API

### 方案 C：事件总线 + Zustand（选中）
- **优点**：实时事件用事件总线，状态共享用Zustand，兼顾解耦性和可维护性
- **缺点**：需要维护两套通信机制

## 理由

1. **分层通信**：实时事件（战斗伤害、技能释放）需要即时响应，适合事件总线；状态共享（角色属性、AI配置）需要持久化，适合Zustand

2. **解耦性**：事件总线实现了发布者和订阅者的完全解耦，引擎和组件可以独立开发和测试

3. **可追踪性**：Zustand提供了清晰的状态管理，便于调试和状态恢复

4. **兼容性**：原生JS引擎无需修改即可与React组件通信

## 影响

### 正面影响
- 核心引擎保持稳定，不受UI框架迁移影响
- 组件可以独立开发和测试
- 通信机制清晰，易于理解和维护

### 负面影响
- 需要维护事件和状态的一致性
- 调试时需要追踪事件流和状态变化
