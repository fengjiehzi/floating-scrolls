# 0003: 全量采用 TypeScript

## 状态

已采纳

## 上下文

「万卷浮生」前端重构项目采用 React + Vite + Zustand + Tailwind CSS 技术栈。需要决定是否引入 TypeScript，以及引入的方式和范围。

## 决策

全量采用 TypeScript，所有新文件都使用 .ts/.tsx 扩展名，配置完整的类型检查。

## 备选方案

### 方案 A：全量 TypeScript（选中）
- **优点**：类型安全、IDE支持好、重构更安全、代码可维护性高
- **缺点**：学习曲线、配置复杂度、构建时间增加

### 方案 B：渐进式引入
- **优点**：降低门槛、快速迭代
- **缺点**：后期迁移工作量大、类型不一致、开发体验分裂

### 方案 C：仅类型声明
- **优点**：平衡类型安全和开发效率
- **缺点**：需要维护两套文件、类型声明可能滞后

## 理由

1. **代码质量**：TypeScript 的类型检查能在编译期发现大量错误，减少运行时 bug

2. **开发体验**：IDE 提供更精准的智能提示、自动补全和重构支持

3. **团队协作**：类型定义作为文档，降低新成员的理解成本

4. **技术栈一致性**：React 生态对 TypeScript 的支持非常成熟，Zustand、React Router、Tailwind CSS 等都提供完整的类型支持

5. **长期维护**：随着项目规模增长，TypeScript 的优势会越来越明显

## 影响

### 正面影响
- 代码质量显著提升
- 开发效率提高（智能提示、自动重构）
- 团队协作更顺畅
- 减少运行时错误

### 负面影响
- 学习曲线较陡
- 配置复杂度增加
- 构建时间略有增加
- 需要编写类型定义

## 实施策略

1. 配置 tsconfig.json，启用严格模式
2. 定义核心类型（角色、战斗、剧情等）
3. 逐步迁移现有 JavaScript 代码到 TypeScript
4. 为原生 JS 引擎添加类型声明文件

## 后续行动

1. 初始化 TypeScript 配置
2. 创建核心类型定义文件
3. 编写类型声明文件（.d.ts）供原生 JS 引擎使用
4. 逐步迁移组件和工具函数到 TypeScript
