# 第一篇：战略视野

## 第一章：产品定位与核心哲学

### 1.1 一句话产品定义

> **万卷浮生——让玩家进入小说，养成角色，改写命运。**

20字以内的核心定义：
> **「进小说，养成角，改命运」**

### 1.2 产品核心身份

**身份：游戏（选A）**

万卷浮生的核心是**小说世界叙事养成与对战**。AI 造卡、小说解析和世界书生成是降低内容生产门槛的手段，最终目标是：

```
进入小说世界 → 养成角色 → 改写剧情 → 验证强弱 → 分享与传播
```

**为什么不是"工具"或"社区"**：
- 工具型产品用户用完即走，没有持续动力
- 社区型产品需要先有大量内容，MVP阶段难以实现
- **游戏是唯一能让用户愿意每天回来、持续投入、并主动分享的形态**

### 1.3 差异化壁垒——"体验链路"，不是单点算法

万卷浮生的真正壁垒**不是**单点AI算法，而是**完整体验链路**：

| 单点能力 | 容易被复制吗 | 我们的优势 |
|----------|--------------|------------|
| AI 提取角色 | ✅ 容易被复制 | 大厂接入更好模型即可 |
| 战力量化 | ✅ 容易被复制 | 算法是公开的 |
| PVP 战斗引擎 | ✅ 容易被复制 | 战斗系统是标准能力 |
| **完整体验链路** | ❌ 难以复制 | 需要产品哲学 + 多模块协同 |
| **用户创造的新剧情被解析和反馈** | ❌ 难以复制 | 需要长期数据积累 |
| **玩家对剧情走向的归属感** | ❌ 难以复制 | 是情感和社交资产 |

**关键判断**：
> 单点 AI 算法容易被追上，但如果用户能持续创造新剧情、养成角色、验证强弱，并且看到系统对自己创造剧情的解析和反馈，**这套体验更难复制**。

### 1.4 古籍活化的战略定位

**古籍活化是重要加分项，不是产品限制。**

| 维度 | 立场 |
|------|------|
| 当前选择 | 优先使用古籍和名著 |
| 比赛契合 | 古籍活化能展示文化价值，契合大赛 |
| 长期方向 | **不只做名著，以小说为主** |
| 战略意义 | 证明经典文本可以被重新进入、重新演绎、重新游玩 |

**古籍活化的三方面公益价值**：

1. **经典文学的新生触达** ——《红楼梦》《西游记》《三国演义》《聊斋》《封神演义》等古籍经典以"收集对战"方式触达新一代读者。玩家为养强孙悟空，会主动去读《西游记》了解技能来源，让古籍从"课本必读"变成"游戏必查"。
2. **古籍角色数字化建档** —— AI 提取过程本身即是对古籍角色的一次结构化数字化：姓名、身份、性格、关系网、关键经历、能力描写均以 JSON 结构存储，形成可检索的古籍角色数据库。
3. **降低古籍阅读门槛** —— 通过"养成 → 查原著 → 再养成"的循环，引导玩家主动阅读古籍原文。MVP 版本预置四大名著角色库，上线后逐步扩展至唐宋传奇、明清小说、外国公版名著。

### 1.5 产品的"四不像"反而是优势

| 常见品类 | 万卷浮生的位置 | 借鉴之处 |
|----------|---------------|----------|
| 文字冒险游戏 | ✅ 核心 | 剧情分支、选择影响、养成感 |
| 卡牌对战游戏 | ✅ 核心 | 角色卡、技能、数值平衡 |
| AI 工具 | ✅ 工具 | 小说解析、角色生成 |
| 内容社区 | ✅ 平台 | UGC 剧情、战报分享 |
| 桌游/跑团 | ✅ 借鉴 | 公开规则、骰子裁决、戏剧性 |

**产品哲学**：
> 万卷浮生**不模仿任何单一产品**，而是把"小说沉浸 + 角色养成 + 自动推演 + 叙事解释"四个独立领域的能力，**整合成一条独特体验链路**。

## 第二章：目标用户与需求分析

### 2.1 用户结构与优先级

| 优先级 | 用户类型 | 占比 | 早期打动力 | 系统要求 |
|--------|---------|------|-----------|----------|
| **P0 核心** | 小说/动漫粉丝 | 40% | 强（对角色有感情） | 角色还原度、剧情代入感 |
| **P1 次核心** | 战力讨论爱好者 | 30% | 中（喜欢讨论强弱） | 战斗可解释、结果可复盘 |
| **P2 潜力** | 内容创作者 | 20% | 强（需要素材） | 可分享的战报、可二创的剧情 |
| **P3 边缘** | 卡牌对战玩家 | 10% | 弱（对平衡要求高） | 平衡性、深度系统 |

**关键判断**：
- **早期最能被打动的是对角色和剧情有感情的人**（小说/动漫粉丝）
- 卡牌对战玩家对平衡和系统深度要求更高，**不应作为最早核心用户**
- 战力讨论爱好者是天然传播者，会主动把"孙悟空vs迪迦"结果分享出去

### 2.2 详细用户画像

#### 用户一：战力讨论爱好者（核心用户）

- **年龄**：15-35 岁
- **特征**：活跃在 B 站、贴吧、知乎的"XX vs YY"战力对比话题下，渴望"真打一场"见分晓
- **行为**：观看战力对比视频、参与评论区辩论、收藏战力排行榜
- **需求**：科学量化的战力数据 + 可对战平台
- **付费意愿**：中（为本命角色变强付费）

#### 用户二：卡牌对战玩家（次核心用户）

- **年龄**：18-30 岁
- **特征**：喜欢《阴阳师》《Fate/GO》《崩坏：星穹铁道》，但苦于官方角色产出慢、想要的角色官方不出
- **行为**：长期玩 1-2 款卡牌游戏、关注角色平衡性、参与 PVP 排位
- **需求**：更多角色选择 + 公平 PVP 环境
- **付费意愿**：高（卡牌游戏用户付费习惯成熟）

#### 用户三：小说/动漫重度粉丝

- **年龄**：16-28 岁
- **特征**：对某些角色有强烈情感，想"拥有"这个角色并培养他变强
- **行为**：阅读原著、参与同人创作、收藏角色周边
- **需求**：把喜欢的角色变成可养成卡牌
- **付费意愿**：中高（为情感付费）

#### 用户四：直播/短视频内容创作者

- **年龄**：20-35 岁
- **特征**：需要新鲜互动内容，"我用 AI 让孙悟空和迪迦打了一场"自带传播潜力
- **行为**：B 站、抖音、YouTube 内容创作，关注热点话题
- **需求**：可生成对战视频/战斗叙述的内容工具
- **付费意愿**：低（但带来大量免费流量）

### 2.3 种子用户来源（开源项目版）

| 渠道 | 类型 | 获取方式 | 吸引点 |
|------|------|----------|--------|
| **GitHub** | P0 | README、Demo、技术文章 | 可 fork、二创、技术展示 |
| **开源社区** | P0 | V2EX、掘金、CSDN | AI + 游戏化完整链路 |
| **AI 应用社区** | P0 | Prompt 调优社区、AI 工具圈 | 小说解析、角色生成 prompt |
| **开发者社群** | P1 | Discord、QQ 群 | 可部署、本地运行 |
| **B站/知乎** | P2 | 演示视频、科普文章 | 跨作品战力讨论话题 |

**核心策略**：
> 早期不做大众推广，**先把项目做成可部署、可二创、可试玩的开源项目**。通过 README、Demo、技术文章和开源传播吸引第一批种子用户。

### 2.4 典型使用场景

**场景一：朋友战力争论**
和朋友吵"孙悟空 vs 迪迦谁厉害"吵不出结果 → 各自上传小说养成角色，PVP 见分晓，输的请客。

**场景二：本命角色养成**
喜欢某个小说角色（如《红楼梦》林黛玉），官方游戏里没有 → 自己上传小说，AI 生成角色卡牌，开始剧情养成。

**场景三：直播互动内容**
直播时需要互动内容 → 观众投票选择出战角色，AI 自动战斗，弹幕押注，AI 生成战斗小说式叙述增加观赏性。

**场景四：跨作品梦幻对决**
想看"诸葛亮 vs 梅林""鲁智深 vs 路飞"→ 上传双方原作小说，AI 量化战力，自动战斗生成结果与叙述。

**场景五：古籍角色收集**
为收集强力 S 级角色，主动阅读《西游记》《封神演义》等古籍 → 在游戏中解锁孙悟空、姜子牙等角色卡牌。

### 2.5 痛点与解决方案对照表

| 痛点 | 现状 | 万卷浮生解决方案 |
| --- | --- | --- |
| 跨作品战力讨论无解 | 只能嘴上吵，永远没结论 | AI 量化战力 + PVP 自动战斗，用结果说话 |
| 卡牌游戏角色受限 | 官方出什么玩家玩什么，想要的角色官方不出 | 玩家上传任意公版小说，AI 提取角色生成卡牌 |
| 小说角色无法"拥有" | 读完小说只能记住角色，无法"养成"他 | 把角色变成可养成卡牌，升级、学技能、解锁形态 |
| 战力对比无科学依据 | 各作品设定不同，无法直接对比 | AI 用统一 7 维度量化所有角色，数值有原作依据 |
| 古籍阅读门槛高 | 课本必读，年轻读者无兴趣 | 游戏化收集对战，引导主动阅读原著 |
| 直播内容同质化 | 缺乏新鲜互动内容 | AI 跨作品对战 + 战斗叙述，自带话题流量 |

### 2.6 用户动机的深度分析

#### 2.6.1 留下来——什么是"第二天还来"的真正动力？

| 表面动机 | 深层动机 | 系统设计 |
|---------|---------|----------|
| 数值涨了 | 自己的角色正在成长 | 养成进度可视化 |
| 解锁新技能 | 系统理解我的角色 | 解锁提示带叙事 |
| 打完一场 | 验证昨天的选择是否对 | 战斗结果关联选择 |
| **看到剧情分支** | **我创造的新剧情正在被理解** | **世界书实时反映玩家行为** |

**用户留下来的真正关键**：
> 第二天回来，不只是为了看数值涨了，而是想知道自己上一轮选择会如何影响剧情、人物关系、世界状态和后续命运。系统要让用户感到：我的角色正在成长，我创造的新剧情正在被理解，后面还有新的分支、新的关系、新的战斗和新的结局。

#### 2.6.2 分享出去——什么会让人主动传播？

| 分享动机 | 触发场景 | 传播效果 |
|---------|---------|----------|
| **我改写的剧情走向** | 玩家在分支中选到离谱/精彩/意外剧情 | 长尾传播，引发讨论 |
| **谁更强的讨论** | 跨作品对战结果意外 | 争议驱动转发 |
| AI 造卡 | "AI居然把XX做出来了" | 单次惊叹 |
| 邀请对战 | "快来和我打一场" | 单次邀请 |

**最强分享动机**：
> 用户会分享自己改写的离谱、精彩或意外剧情，也会分享角色对战结果，**引发朋友讨论这个剧情是否合理、这个角色是否应该更强、系统给出的胜负解释是否说得通**。

## 第三章：市场分析与机会窗口

### 3.1 行业背景

**AI + 游戏趋势：**
- 2024-2026 年 AIGC 游戏方向爆发，AI 已能完成长文本理解、角色特征提取、数值平衡、战斗叙述生成等任务，技术条件成熟
- 字节跳动、腾讯、网易均在布局 AI + 游戏赛道，TRAE 大赛主题"世界很大，放手去造"明确鼓励 AI 原生创作
- AI 角色扮演、AI NPC、AI 生成内容（AIGC）成为游戏行业新增长点

**卡牌 PVP 游戏市场：**
- 《阴阳师》运营 8 年仍盈利，验证卡牌 PVP 高粘性商业模式
- 《Fate/GO》《崩坏：星穹铁道》全球年收入超 10 亿美元，市场体量巨大
- 玩家痛点：官方角色产出慢、想要的角色官方不出、跨作品对战无平台

**跨作品战力讨论流量：**
- B 站"孙悟空 vs XXX"类战力对比视频动辄百万播放
- 贴吧"战力吧""跨作品对战吧"活跃用户数十万
- 知乎"XX 和 YY 谁更厉害"问题常年热门，证明用户需求强烈

### 3.2 竞品对比表

| 对比维度 | 阴阳师 | Fate/GO | 崩坏：星穹铁道 | **万卷浮生** |
| --- | --- | --- | --- | --- |
| 角色来源 | 日本阴阳师 IP | Fate 系列 IP | 米哈游原创 | **任意公版小说 + 用户原创** |
| 角色数量 | ~200 | ~300 | ~50 | **近乎无限（AI 提取）** |
| 养成方式 | 抽卡 + 升级 + 御魂 | 抽卡 + 升级 + 礼装 | 抽卡 + 升级 + 遗器 | **剧情推进 + 选择分支** |
| PVP 模式 | 同作品内斗技 | 基本无 PVP | 模拟宇宙 PVE | **跨作品 1v1 自动战斗** |
| 战力依据 | 官方数值策划 | 官方数值策划 | 官方数值策划 | **AI 原作还原算法** |
| 创新点 | 日式和风美术 | 剧情向 IP 运营 | 箱庭探索 + 战斗 | **AI 造卡 + 跨作品对战** |
| 玩家创作 | 无 | 无 | 无 | **玩家上传小说造卡** |

### 3.3 四大市场机会

**机会一：AI 技术成熟窗口期**
长文本理解（GLM-4-Flash、Claude、通义千问）+ 结构化输出（JSON Mode）+ 多模型适配层技术均已成熟，AI 提取角色 + 战力量化的成本与准确度达到产品化临界点。

**机会二：跨作品对战蓝海**
现有卡牌游戏均局限于单一 IP 内对战，"孙悟空 vs 迪迦"这类跨作品对战无任何平台满足。万卷浮生是首个用 AI 解决跨作品战力量化与对战的产品。

**机会三：内容供给无限化**
公版小说数量庞大（四大名著、唐宋传奇、明清小说、外国公版名著），每本可提取数十角色，角色卡牌供给近乎无限，彻底打破官方产能瓶颈。

**机会四：天然社交传播**
"孙悟空 vs 迪迦"类话题自带百万级流量，每个对战视频都是免费宣传，获客成本极低，契合短视频时代内容营销。

## 第四章：核心价值主张

### 4.1 核心价值三要素

万卷浮生是一个 Web 端 AI 角色卡牌 PVP 游戏。玩家上传公版小说文本，AI 自动提取角色并按统一维度科学计算战力属性，生成可对战角色卡牌；玩家通过剧情养成角色，再与其他玩家的角色进行 1v1 自动战斗 PVP，用战斗结果回答"孙悟空 vs 迪迦谁更厉害"这类跨作品战力讨论。

**核心价值三要素：**

1. **AI 造卡** —— 任意公版小说 → 几分钟生成可对战角色卡牌，内容供给近乎无限
2. **科学量化** —— 统一 7 维度 + 原作还原算法，跨作品角色科学可比
3. **真打一场** —— 1v1 自动战斗 PVP，把"嘴上吵"变成"用结果说话"

### 4.2 与传统卡牌游戏对比

| 维度 | 传统卡牌游戏（阴阳师/Fate/GO/崩坏星穹铁道） | 万卷浮生 |
| --- | --- | --- |
| 角色来源 | 官方预设，开发者策划 | 玩家上传任意公版小说，AI 提取 |
| 角色数量 | 数十至数百，受产能限制 | 近乎无限，公版小说库 + 用户原创 |
| 战力依据 | 官方数值策划设定 | AI 根据原作描写科学计算，有据可查 |
| 养成方式 | 抽卡 + 升级 + 装备 | 推进原作剧情，选择影响成长方向 |
| PVP 模式 | 同作品内角色对战 | 跨作品梦幻对决（孙悟空 vs 迪迦） |
| 创作门槛 | 玩家无法创造角色 | 任何玩家几分钟"造"出一批卡牌 |

### 5.2 模块详解

#### 模块 1：小说解析引擎

**核心能力**：

| 能力 | 实现方式 | 准确率预期 |
|------|----------|-----------|
| 文本分章节 | 按章节标记切分 | 95%+ |
| 角色识别 | NLP 提取 + 大模型校验 | 70-80% |
| 关系抽取 | 大模型 + 规则 | 60-70% |
| 战力描写识别 | 规则 + 关键词 | 50-60% |
| 世界观总结 | 大模型生成 | 80-90% |

**输入**：公版小说（txt / epub / 粘贴）
**输出**：世界书 JSON + 角色卡候选列表

**MVP 限制**：
- 优先支持四大名著 + 经典小说（公版）
- 长文本分块处理
- 解析结果需要玩家确认

#### 模块 2：AI 角色生成器（双轮提取流程）

**核心能力**：

```
┌─────────────────────────────────────────┐
│  第一轮 · 角色卡提取（Character Card）   │
│                                         │
│  输入：小说全文（分章节送入 AI）          │
│  AI 任务：逐章扫描，提取每个角色的：     │
│    - 姓名、身份、性格、关系网            │
│    - 关键经历、所有战斗/能力描写         │
│  输出：角色卡 JSON（结构化文本）          │
└─────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│  第二轮 · 战力量化（Power Quantization） │
│                                         │
│  输入：第一轮生成的角色卡                │
│  AI 任务：根据能力描写，按统一 7 维度    │
│          计算战力数值，生成分级与技能    │
│  输出：可对战角色卡牌 JSON（含数值）     │
└─────────────────────────────────────────┘
```

**为什么分两轮？**
- 第一轮专注"理解角色"，避免数值计算干扰角色理解
- 第二轮专注"量化战力"，基于完整角色卡做数值计算，结果更准确
- 玩家可查看"为什么孙悟空力量值是 98"，依据来自第一轮的原作描写引用

**设计哲学**：
> AI 造卡失败时，采用"AI 初稿 + 玩家编辑修正"的方式。AI 负责生成基础角色、技能和数值，玩家可以修正不准的地方，系统再做基础校验和标记。这样既承认 AI 可能出错，也让玩家参与到角色共创里。

**数值校验规则**：
- 等级 S/A/B/C 对应数值范围
- 技能不能超过 5 个
- 形态不超过 3 个
- HP/MP 在合理区间

#### 模块 4：角色养成系统

**核心机制**：

| 养成维度 | 玩家操作 | 系统反馈 |
|----------|----------|----------|
| 等级 | 战斗胜利获得经验 | 自动升级，解锁新内容 |
| 经验 | 参与剧情、使用角色 | 累积到下一级 |
| 技能升级 | 选择技能投入资源 | 数值提升，叙事更新 |
| 形态解锁 | 推进剧情到关键节点 | 形态可用，战斗前可切换 |
| 剧情选择 | 在节点图中做选择 | 影响属性成长方向 |

**养成路线设计**：
- **攻击路线**：偏力量、特殊能力，解锁"大闹天宫"类形态
- **防御路线**：偏防御、智力，解锁"斗战胜佛"类形态
- **均衡路线**：全属性+少量加成，解锁多个形态

**关键设计原则**：
> 同一角色可多次养成（不同存档），探索不同路线。每次养成都是新故事。

#### 模块 5：战前策略系统

**玩家在战前的选择**：

```
出战角色选择
   ↓
形态选择（基础/进阶/终极）
   ↓
技能配置（5选3或类似）
   ↓
战术倾向（进攻/防守/平衡）
   ↓
装备/法宝（可选）
   ↓
确认 → 进入战斗推演
```

**关键设计**：
> 战前策略让玩家有参与感，但战中由系统自动推演。这样既有参与感，又不需要做复杂的即时操作，也更适合保留 AI 推演和叙事解释的特色。

#### 模块 6：战斗推演引擎

**核心实现**（基于现有 battle-engine.js）：

```
回合循环（最多12回合）：
   1. 按速度决定先手
   2. AI 选技能（基于血量/MP/技能列表）
   3. 伤害计算
      base = 属性 × 1.2
      damage = base × skillMult × critMult × defReduction × gradeMult × dimensionMult
   4. 判定胜负
   5. 清理 buff
   6. 进入下一回合
```

**关键公式**（已实现）：

| 公式 | 计算方式 |
|------|----------|
| 暴击率 | `min(0.35, (speed + intelligence) / 400)` |
| 防御减免 | `1 - (defense / (defense + 200))` |
| 等级压制 | S>A>B>C，差1级 1.5x，差2级 2.0x |
| 次元压制 | 神话打凡人 3.0x，凡人打神话 0.5x |
| 秒杀判定 | 伤害 > 目标 HP × 60% 触发⚡秒杀 |

**平衡 vs 真实的处理**：
> 这个游戏不需要太追求传统意义上的平衡性。更重要的是有一套公开、一致的基准规则。在同一基准下，强弱差异应该自然体现出来，这样才符合跨作品战力讨论的核心乐趣。只要规则透明、标准一致，强角色就应该强，弱角色也可以通过特定条件、战术或剧情解释获得少量机会，但不需要强行五五开。

#### 模块 7：叙事解释系统

**这是产品的灵魂模块**——战斗不只是结算胜负，而是要把胜负过程讲得精彩、合理、有戏剧性。

**叙事生成的三个层次**：

1. **回合级**："孙悟空施展筋斗云突袭，瞬间逼近迪迦，造成152点伤害"
2. **关键节点级**："第三回合，迪迦切换强力型，防御骤然提升，孙悟空的连击效果骤降"
3. **胜负级**："经过5回合激战，孙悟空以微弱优势险胜。这场对决的胜负关键在于第二回合孙悟空抢到了先手，打出了关键暴击……"

**秒杀局叙事**：
> 这一击挟神话级威能，凡人之躯难以承受。关羽虽勇，终究无法跨越次元的鸿沟。
> ⚡ 仅1回合，孙悟空便以碾压之势秒杀关羽！

**核心哲学**：
> 核心爽点是叙事解释。数值只是底层结果，真正打动用户的是系统如何解释角色为什么赢、为什么输、关键回合为什么发生。让用户觉得"这个结果说得通，甚至值得转发"。

### 6.2 链路设计原则

#### 原则 1：每个环节都要"服务叙事"
- AI 解析 → 帮用户理解小说的世界
- 角色生成 → 让用户感觉"我的角色"被创造
- 养成 → 让用户感觉"我的角色在成长"
- 战斗 → 让用户感觉"我的角色的强弱被验证"
- 分享 → 让用户感觉"我的故事值得被传播"

#### 原则 2：用户的创造要被系统"理解"和"反馈"
- 用户在剧情分支中的选择 → 影响世界书
- 用户对角色的微调 → 影响后续推演
- 用户的战斗结果 → 关联回剧情

**核心承诺**：
> 你创造的新剧情正在被理解，后面还有新的分支、新的关系、新的战斗和新的结局。

#### 原则 3：保留"轻量化入口"
- 用户不必从零开始
- 预置角色库（14 个 + 持续扩展）让用户可以快速体验
- 选择"用预置角色"vs"上传小说 AI 解析"两条路径

### 6.3 MVP 阶段的核心闭环

```
公版小说 → AI 解析 → 角色卡列表 → 选择角色 → 战前策略 → 战斗推演 → 叙事解释 → 分享战报
                                              ↑
                                              └── 回到角色库（换角色再来）
```

**MVP 范围说明**：
- 第一版只跑通这条最小闭环
- 养成系统是简化版（只支持等级和形态）
- 剧情选择是预置的（不做动态生成）
- 但**叙事解释**必须做到位

### 6.4 核心循环

```
上传公版小说
      ↓
AI 双轮提取（角色卡 + 战力量化）
      ↓
生成可对战角色卡牌
      ↓
剧情养成（推进节点 → 选择分支 → 成长/解锁技能）
      ↓

      ↓

      ↓

```

### 6.5 玩家旅程说明

**第一阶段：造卡（首次体验）**
1. 玩家进入首页，选择"上传小说"或"使用预置角色库"
2. 上传公版小说文本（支持 .txt / .epub / 粘贴文本）
3. AI 在 1-3 分钟内完成双轮提取，生成角色卡牌列表
4. 玩家查看角色卡牌，了解属性、技能、原作依据
5. 选择一个本命角色作为初始角色

**第二阶段：养成（深度体验）**
1. 进入角色剧情有向节点图
2. 推进剧情节点，遇到选择分支
3. 不同选择影响角色成长方向（攻击型/防御型/智慧型）
4. 关键节点解锁新技能或形态变化
5. 重复养成多个角色，构建角色库

**第三阶段：对战（核心价值）**
1. 选择一个角色出战
2. 系统匹配对手（跨作品角色）
3. 自动战斗：双方按速度决定出手顺序，属性计算伤害，技能触发
4. AI 生成战斗小说式叙述
5. 查看战斗回放、属性对比、胜负原因分析
6. 获得排名变化与奖励

**第四阶段：社交传播（裂变）**
1. 分享战斗回放至 B 站/抖音
2. 邀请朋友上传小说对战
3. 参与跨作品对战话题

### 6.6 玩法特色

1. **AI 造卡，内容无限** —— 任意公版小说 → 几分钟生成可对战角色卡牌，打破官方产能瓶颈
2. **科学量化，跨作品可比** —— 统一 7 维度 + 原作还原算法，孙悟空"筋斗云十万八千里"→速度拉满，数值有据可查
3. **剧情养成，选择影响成长** —— 推进原作剧情，选择"大闹天宫"还是"保护唐僧"影响角色成长方向
4. **跨作品梦幻对决** —— 孙悟空 vs 迪迦、诸葛亮 vs 梅林，让永远吵不出结果的战力讨论真正打一场
5. **AI 战斗叙述** —— 根据战斗数据生成小说式叙述，增加观赏性与传播力

## 第八章：游戏系统设计

### 8.1 战力量化系统

#### 8.1.1 统一 7 维度属性

所有角色（无论修仙/武侠/玄幻/科幻/现实）统一提取为以下 7 维度属性：

| 维度 | 英文 | 含义 | 数值范围 |
| --- | --- | --- | --- |
| 力量 | power | 物理攻击强度与爆发力 | 0-100 |
| 速度 | speed | 移动速度与出手频率 | 0-100 |
| 智力 | intelligence | 战术策略与心理素质 | 0-100 |
| 防御 | defense | 生存能力与抗打击能力 | 0-100 |
| 特殊能力 | special_ability | 法术/异能/特殊技能强度 | 0-100 |
| 生命值 | hp | 最大生命值 | 2500-9500 |
| 法力值 | mp | 最大法力值 | 300-1500 |

**衍生属性（由 7 维度计算得出）：**

| 衍生属性 | 计算公式（实际代码） |
| --- | --- |
| 暴击率 | `crit_rate = (speed + intelligence) / 400`，上限 35% |
| 防御减免 | `def_reduction = 1 - (defense / (defense + 200))` |

#### 8.1.2 原作还原算法说明

AI 根据原作描写科学计算数值，核心原则：**原作描写 → 数值映射 → 引用可查**。

**映射规则示例：**

| 原作描写 | 维度 | 数值映射 | 依据 |
| --- | --- | --- | --- |
| 孙悟空"一个筋斗十万八千里" | 速度 | 99 | 超光速级位移，速度拉满 |
| 鲁智深"倒拔垂杨柳" | 力量 | 92 | 拔起大树需数吨力量 |
| 诸葛亮"空城计" | 智力 | 98 | 心理战巅峰，智力拉满 |
| 迪迦"复合型形态" | 全维度 | 88-92 | 均衡发展型战士 |
| 林黛玉"弱柳扶风" | 防御 | 20 | 体弱多病，防御极低 |
| 姜子牙"封神榜" | 特殊能力 | 98 | 封神级法术 |

#### 8.1.3 分级平衡机制

角色按原作强度分为 S/A/B/C 四级，**同级内平衡，跨级保留原作差距**：

| 等级 | 数值映射 | 标准 | 代表角色 |
| --- | --- | --- | --- |
| S 级 | 4 | 神级/超越物理法则 | 孙悟空、迪迦、姜子牙、后羿、二郎神 |
| A 级 | 3 | 超凡/远超常人 | 鲁智深、关羽、哪吒、白素贞、项羽 |
| B 级 | 2 | 强者/明显强于常人 | 武松、林黛玉、貂蝉 |
| C 级 | 1 | 常人/略强于常人 | 普通士兵、路人角色 |

**等级压制倍率（实际代码）：**

```javascript
// battle-engine.js 第 29-39 行
function getGradeMult(attacker, defender) {
    const atkVal = GRADE_VALUE[attacker.grade] || 2;
    const defVal = GRADE_VALUE[defender.grade] || 2;
    const diff = atkVal - defVal;
    if (diff >= 2) return 2.0;   // S打B：2倍
    if (diff === 1) return 1.5;  // S打A 或 A打B：1.5倍
    if (diff <= -2) return 0.5;  // B打S：减半
    if (diff === -1) return 0.8; // A打S 或 B打A：8折
    return 1.0;
}
```

**平衡原则：**
- **同级平衡**：S 级角色间总战力值接近，保证 PVP 公平
- **跨级保留差距**：S 级 vs B 级保留原作差距（2倍伤害）
- **PVP 匹配优先同级**：匹配系统优先匹配同级角色，跨级对战需玩家确认

| "保护唐僧"路线 | 防御与智慧成长快（defense +4, intelligence +4/节点） | 斗战胜佛形态、心理战技能 |
| "兼修路线" | 均衡成长（全属性 +2/节点） | 两种形态部分解锁 |

**成长机制设计：**
- 每个节点选择带来 2-5 点属性成长
- 关键节点解锁技能或形态
- 同一角色可多次养成（不同存档），探索不同路线
- 养成完成后角色可参与 PVP

#### 8.2.3 技能/形态解锁机制

**技能解锁条件：**
- 剧情节点解锁：推进到特定剧情节点自动解锁（如孙悟空推进至"拜师菩提"解锁七十二变）
- 属性阈值解锁：属性达到一定值解锁（如智力 ≥ 80 解锁"心理战"技能）
- 多周目解锁：完成一次养成后，新周目可解锁隐藏技能

**形态解锁条件：**
- 关键剧情节点解锁（如孙悟空"大闹天宫"节点解锁齐天大圣形态）
- 形态切换在 PVP 前选择，PVP 中不可切换
- 每个形态提供不同属性加成与技能组合

#### 8.2.4 多角色收集机制

- 玩家可从多本小说收集多个角色
- 每个角色独立养成，有独立存档
- PVP 时只能选一个角色出战
- 角色库支持分类筛选（按小说/等级/属性）
- 收集成就系统：收集某小说全部角色解锁特殊称号

### 8.3 PVP 自动战斗系统

#### 8.3.1 战斗回合流程

```javascript
// 最大回合数：12回合
const maxRounds = 12;

// 按速度决定先手
const firstAttacker = char1.stats.speed >= char2.stats.speed ? 1 : 2;

// 战斗循环
while (round <= maxRounds && char1.currentHp > 0 && char2.currentHp > 0) {
    // 先手攻击
    const action = executeAttack(battle, attacker, defender, atkSide, defSide);
    // 判定胜负
    if (char2.currentHp <= 0) { battle.winner = 1; break; }
    // 后手攻击
    const action = executeAttack(battle, attacker, defender, atkSide, defSide);
    // 判定胜负
    if (char1.currentHp <= 0) { battle.winner = 2; break; }
    // 回合结束清理buff
    endTurnBuffs(battle);
    round++;
}

// 超时判定：按HP比例判定胜负
if (!battle.winner && round > maxRounds) {
    const hp1Ratio = char1.currentHp / char1.stats.hp;
    const hp2Ratio = char2.currentHp / char2.stats.hp;
    battle.winner = hp1Ratio > hp2Ratio ? 1 : 2;
}
```

#### 8.3.2 属性计算公式

**伤害计算公式（实际代码）：**

```javascript
// 物理攻击伤害
if (skill.type === 'physical_attack') {
    base = attacker.stats.power * 1.2;
}
// 魔法攻击伤害
else if (skill.type === 'magic_attack') {
    base = attacker.stats.special_ability * 1.2;
}
// 速度攻击伤害
else if (skill.type === 'speed_attack') {
    base = attacker.stats.speed * 1.2;
}
// 召唤分身伤害
else if (skill.type === 'summon') {
    base = attacker.stats.power * 0.6;
}

// 加上特殊能力加成
base += attacker.stats.special_ability * 0.3;

// 最终伤害计算
damage = base * skillMult * critMult * defReduction * attackDownMult * gradeMult * dimensionMult;
```

**公式详解：**

| 参数 | 计算方式 | 说明 |
| --- | --- | --- |
| base | 力量×1.2 或 特殊能力×1.2 或 速度×1.2 | 基础伤害 |
| skillMult | 技能倍率（1.0-2.0） | 技能伤害倍率 |
| critMult | 暴击1.5，非暴击1.0 | 暴击倍率 |
| defReduction | `1 - (defense / (defense + 200))` | 防御减免 |
| attackDownMult | 0.7（有debuff）或 1.0 | 攻击降低 |
| gradeMult | 等级压制倍率（见5.2.3） | 等级压制 |
| dimensionMult | 次元压制倍率（见5.4.4） | 次元压制 |

#### 8.3.3 技能触发逻辑

```javascript
function selectSkill(character, currentHp, currentMp) {
    const hpRatio = currentHp / character.stats.hp;
    // 可用技能（排除被动，MP足够）
    const availableSkills = character.skills.filter(s =>
        s.type !== 'passive' && currentMp >= s.mp_cost
    );

    // MP不足时使用普攻
    if (availableSkills.length === 0) {
        return character.skills.find(s => s.mp_cost === 0 && s.type !== 'passive');
    }

    // 血量低于30%优先防御/回血
    if (hpRatio < 0.3) {
        const defensive = availableSkills.find(s => s.type === 'transform');
        if (defensive && Math.random() < 0.5) return defensive;
        const healSkill = availableSkills.find(s => s.type === 'heal');
        if (healSkill && Math.random() < 0.6) return healSkill;
    }

    // MP充足时优先高伤害技能（70%概率选最高倍率）
    const attackSkills = availableSkills.filter(s =>
        ['physical_attack', 'magic_attack', 'speed_attack', 'summon'].includes(s.type)
    );
    attackSkills.sort((a, b) => b.multiplier - a.multiplier);
    if (Math.random() < 0.7) return attackSkills[0];
    return attackSkills[Math.floor(Math.random() * attackSkills.length)];
}
```

**技能类型：**

| 类型 | 英文 | 伤害依据 | MP消耗 |
| --- | --- | --- | --- |
| 物理攻击 | physical_attack | 力量×1.2 | 0-80 |
| 魔法攻击 | magic_attack | 特殊能力×1.2 | 60-150 |
| 速度攻击 | speed_attack | 速度×1.2 | 20-60 |
| 变化技能 | transform | 无伤害，增益效果 | 30-100 |
| 回血技能 | heal | 恢复HP 20% | 80-100 |
| 召唤分身 | summon | 力量×0.6 | 60-120 |
| 被动技能 | passive | 自动生效 | 0 |

#### 8.3.4 次元压制机制

```javascript
// 神话类作品
const MYTH_SOURCES = ['西游记', '封神演义', '山海经', '白蛇传', '奥特曼'];

function isMythic(character) {
    return MYTH_SOURCES.includes(character.source);
}

// 次元压制倍率
function getDimensionMult(attacker, defender) {
    const atkMythic = isMythic(attacker);
    const defMythic = isMythic(defender);
    // 神话打凡人：3倍伤害（秒杀级）
    if (atkMythic && !defMythic) return 3.0;
    // 凡人打神话：0.5倍伤害（几乎无效）
    if (!atkMythic && defMythic) return 0.5;
    // 同类对决：正常
    return 1.0;
}
```

**次元压制效果表：**

| 攻方类型 | 守方类型 | 倍率 | 效果 |
| --- | --- | --- | --- |
| 神话（西游记） | 凡人（三国演义） | 3.0 | 秒杀级伤害 |
| 神话（封神演义） | 凡人（史记） | 3.0 | 秒杀级伤害 |
| 凡人（水浒传） | 神话（西游记） | 0.5 | 几乎无效 |
| 神话 vs 神话 | - | 1.0 | 正常对决 |
| 凡人 vs 凡人 | - | 1.0 | 正常对决 |

**设计理念：**
- 神话级角色（孙悟空、哪吒、姜子牙）对凡人级角色（关羽、鲁智深、项羽）形成碾压
- 体现"金箍棒一万三千五百斤"对凡人躯体的毁灭性
- 凡人打神话几乎无效，体现"人力难敌神力"

| 长文本解析 | 用户上传百万字小说 | 分块、并发、缓存 |
| 大模型调用 | 高峰期排队 | 队列、限流、异步 |
| 批量生成 | 一次性造 10 个角色 | 增量、缓存 |
| 成本失控 | 大量用户同时使用 | 限额、降级策略 |

**次要瓶颈**：
- 数据库写入（采用 WAL 模式已优化）
- 实时连接（WebSocket 状态管理）
- 前端展示（CSS 动画、SVG 优化）

**未来桌面版的优势**：
- 本地模型调用（零网络成本）
- 本地缓存（不依赖云端）
- 离线运行（不依赖网络）

### 10.5 数据安全与版权

| 风险 | 当前策略 | 未来桌面版 |
|------|----------|------------|
| 版权问题 | 优先使用公版/授权；用户上传默认私有 | 全部本地存储，不上传云端 |
| 隐私问题 | 数据绑定用户；最小化云端存储 | 默认本地，零云端传输 |
| 内容安全 | 基础限制 + 举报/删除机制 | 不涉及公开传播 |
| 公开分享 | 需用户主动确认 + 基础审核 | 用户自主选择分享方式 |

**关键判断**：
> 公开展示优先使用公版小说、古籍或授权内容；用户上传内容默认私有，不进入公共内容池。未来桌面版应优先本地保存，减少不必要的云端传输。

**公版小说界定标准：**
- 中国作品：作者逝世 50 年以上的作品（如四大名著、《聊斋》《封神演义》）
- 外国作品：符合《伯尔尼公约》公版标准
- 用户原创：用户上传自己拥有版权的作品
- 不支持：仍在版权期内的商业作品（如《斗破苍穹》《三体》等）

### 14.2 核心风险详解

#### 风险 1：AI 造卡质量（最大风险）

**问题描述**：
- 纯规则/NLP 准确率只有 40-60%
- 玩家对"本命角色被识别错"极度敏感
- 大模型可能产生不存在的技能或数值

**应对策略**：

```
策略 A：AI 初稿 + 玩家修正（已确定）
  - AI 负责生成基础内容
  - 玩家可以修改描述、技能、数值
  - 系统做基础校验

策略 B：多模型集成
  - 不同模型各取所长
  - 投票/交叉验证

策略 C：玩家共建角色库
  - 优秀修正可分享到公共库
  - 形成 UGC 飞轮
```

**关键判断**：
> 早期接受 AI 不完美，但通过玩家修正机制让用户参与到角色共创中。这反而是产品的特色，不是缺陷。

#### 风险 2：战斗体验（核心风险）

**问题描述**：
- 纯数值战斗可能无聊
- 跨作品对战天然不平衡
- 没有操作感，玩家可能感觉"与我无关"

**应对策略**：

| 策略 | 说明 | 优先级 |
| --- | --- | --- |
| 强化叙事解释 | 让每场战斗都有故事 | P0 |
| 战前策略选择 | 让玩家有参与感 | P0 |
| 可分享战报 | 让战斗结果可传播 | P1 |
| 视觉特效升级 | 让战斗更"好看" | P1 |
| 关键回合慢镜头 | 增强戏剧性 | P2 |

**关键判断**：
> 战斗的核心是叙事解释，不是数值计算。系统要解释"为什么赢、为什么输、关键回合为什么发生"，让用户觉得"这个结果说得通，甚至值得转发"。

#### 风险 3：传播乏力

**问题描述**：
- 项目没人知道
- 早期用户极少
- 难以形成口碑

**应对策略**：

| 阶段 | 策略 |
| --- | --- |
| 比赛期 | 比赛平台曝光 + B站/知乎内容 |
| 比赛后 | GitHub 持续维护 + 开发者社区 |
| 桌面化后 | 多平台发布 + 玩家社区 |

**开源项目的特殊优势**：
- 不依赖单一渠道
- 技术社区自带传播
- fork 和二创带来自然增长

#### 风险 4：桌面重构超预期

**问题描述**：
- Godot 4 学习成本
- 从 Web 到桌面的全面重构
- 工作量可能是 Web 版的 2-3 倍

**应对策略**：

```
分阶段重构：
  阶段 1：Godot 4 原型（验证可行性） 2026-Q4
  阶段 2：核心功能迁移              2027-Q1
  阶段 3：完整产品实现                2027-Q2
  阶段 4：优化与发布                  2027-Q3

可回退设计：
  - Web 版继续维护
  - 桌面版不达预期则保留双版本
  - 保持核心业务逻辑独立
```

### 14.3 技术风险扩展（来自 v1）

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| **AI 提取准确度不足** | 角色卡牌质量参差不齐，影响体验 | 1. 双轮提取提升准确度；2. Prompt 持续优化与 A/B 测试；3. 玩家可手动编辑角色卡；4. 社区投票校验 |
| **长文本处理超限** | 数十万字小说超出上下文窗口 | 1. 分章节送入 AI；2. 跨章节角色合并；3. 长文本摘要后再提取；4. 限制单次上传字数（如 50 万字） |

| **数值膨胀** | 长期运营后数值失控 | 1. 严格 0-100 分制；2. 总战力值区间限制；3. 定期数值重铸 |

| --- | --- | --- |
| **非公版小说上传** | 侵犯版权，法律风险 | 1. **仅支持公版小说**（明确告知用户）；2. AI 自动识别疑似非公版内容并拦截；3. 用户举报机制；4. 用户协议明确版权责任 |
| **用户原创小说版权** | 版权归属争议 | 1. 用户上传原创小说需勾选授权协议；2. 角色卡牌版权归用户所有；3. 平台获得展示与对战授权 |
| **角色立绘版权** | AI 文生图可能涉及版权 | 1. 使用合规 AI 文生图服务；2. 不模仿特定画师风格；3. 商用立绘使用付费服务 |
| **版权审核机制** | 违规内容上线 | 1. 预置角色库人工审核；2. 用户上传内容 AI 初审 + 人工复审；3. 违规内容下架机制 |

### 14.5 风险应对总原则

1. **技术风险**：多模型适配 + 缓存策略 + 持续优化
2. **版权风险**：严格公版限制 + AI 审核 + 用户协议
3. **重构风险**：分阶段推进 + 可回退设计
4. **整体原则**：MVP 快速验证 → 数据驱动迭代 → 长期运营积累

### 14.6 关键判断

> **对于开源项目，最大的风险不是"没人用"，而是"做得太重"。** 保持小而美、持续迭代、文档完善，是开源项目长期健康的根本。

| 维度 | 立场 |

## 📜 结语

万卷浮生的核心不是"AI 造卡 + PVP 对战"，而是：

> **让玩家进入小说，养成角色，改写命运。**

我们相信：
- 用户的创造应该被系统理解和反馈
- 战斗的胜负应该被精彩地解释
- 跨作品的角色应该可以真正"打一场"
- 经典文本应该被重新进入、重新演绎、重新游玩

通过 **完整的体验链路**（小说解析 → 世界书 → 角色养成 → 战前策略 → 战斗推演 → 叙事解释 → 分享），万卷浮生正在构建一个难以复制的独特产品。

**短期**：完成 Web MVP，参加 TRAE 大赛，开源传播
**长期**：重构为 Godot 4 桌面应用，做成本地小说世界工作台

**这不只是一个游戏，而是小说与玩家的新型互动方式。**
