## 一、游戏概念与核心机制

### 1.1 游戏概述

| 项目 | 内容 |
|------|------|
| 游戏名称 | 万卷浮生 |
| 游戏类型 | AI角色卡牌PVP在线对战 |
| 平台 | Web端（响应式，移动端兼容） |
| 核心模式 | 单人剧情养成 + 在线PVP自动战斗 |
| 目标用户 | 国风文化爱好者、卡牌对战玩家、网文/古典小说读者 |
| 商业模式 | 免费+内购（订阅/皮肤/赛事） |
| 开发引擎 | 原生Web技术栈（无游戏引擎） |
| 联网要求 | 在线对战需联网，剧情养成可离线 |

### 1.2 核心玩法循环

万卷浮生的核心玩法循环由五个环节构成，形成"收集—养成—对战—收集"的闭环：

```
上传小说文本 → AI提取角色属性 → 战力量化生成卡牌 → 剧情养成解锁形态 → PVP自动战斗获取奖励
      ↑                                                                    |
      └────────────────────────────────────────────────────────────────────┘
```

**循环详解**：

1. **上传小说**：玩家上传公版小说文本（或选择系统预置角色）
2. **AI提取角色**：系统通过AI双轮提取流程，从文本中识别角色并生成结构化卡牌数据
3. **战力量化**：基于原作事实，将角色能力科学量化为7维度数值
4. **剧情养成**：通过有向节点图剧情关卡，解锁角色技能与形态变化
5. **PVP自动战斗**：在擂台匹配对手，观看AI驱动的自动战斗，获取战绩与奖励

### 1.3 核心机制

万卷浮生围绕四大核心机制构建独特体验：

#### 1.3.1 AI造卡机制

- **双轮提取流程**：第一轮提取角色基本信息与原作事实，第二轮基于事实生成战力数值与技能
- **原作还原**：所有属性数值必须有原作文本依据（如孙悟空金箍棒一万三千五百斤→power: 98）
- **结构化输出**：生成符合标准JSON Schema的角色卡数据

#### 1.3.2 战力量化机制

- **7维度属性**：power（力量）、speed（速度）、intelligence（智力）、defense（防御）、special_ability（特殊能力）、hp（生命值）、mp（法力值）
- **分级平衡**：S/A/B/C四级，各级别有数值区间约束
- **原作事实驱动**：数值差异体现原作设定（如孙悟空power=98 vs 林黛玉power=25）

#### 1.3.3 PVP自动战斗机制

- **纯计算无API调用**：战斗引擎为确定性计算，不依赖外部AI接口
- **最大12回合**：按速度决定先手，每回合双方各行动一次
- **AI选技能**：基于血量、MP、技能倍率的智能选择算法
- **实时叙述生成**：每场战斗生成完整的文学化战斗叙述

#### 1.3.4 次元压制机制（核心差异化）

这是万卷浮生最具特色的机制，体现了"神话级 vs 凡人级"的跨次元差距：

| 攻击方 | 防守方 | 伤害倍率 | 设计意图 |
|--------|--------|----------|----------|
| 神话级 | 凡人级 | **3.0倍** | 体现神话角色的碾压性力量 |
| 凡人级 | 神话级 | **0.5倍** | 凡人难以伤害神话存在 |
| 同类对决 | 同类对决 | 1.0倍 | 正常战斗 |

**神话级作品**：西游记、封神演义、山海经、白蛇传、奥特曼
**凡人级作品**：三国演义、水浒传、红楼梦、史记

**次元秒杀规则**：当神话级攻击凡人级时，伤害至少为目标最大HP的80%，体现"金箍棒一万三千五百斤对凡人躯体的毁灭性"。

### 1.4 差异化定位

与主流卡牌对战游戏对比：

| 对比维度 | 万卷浮生 | 阴阳师 | Fate/GO | 崩坏星穹铁道 |
|----------|----------|--------|---------|--------------|
| 角色来源 | 公版小说AI提取 | 官方原创IP | 官方授权IP | 官方原创IP |
| 造卡方式 | 玩家上传文本AI生成 | 官方固定卡池 | 官方固定卡池 | 官方固定卡池 |
| 战斗模式 | 自动战斗+次元压制 | 半即时回合制 | 指令卡回合制 | 回合制 |
| 跨作品对战 | ✅ 核心特色 | ❌ | ❌ | ❌ |
| 文化属性 | 中国古典文学 | 日本阴阳道 | 多元神话 | 科幻幻想 |
| 平台 | Web端 | 移动端 | 移动端 | 全平台 |
| 开发成本 | 低（AI生成内容） | 高 | 高 | 极高 |

### 1.5 设计哲学

万卷浮生遵循三大设计哲学：

1. **原作还原**：所有角色属性、技能、形态必须有原作文本依据，AI不得凭空创造。每张角色卡附带`source_basis`字段记录原作依据。

2. **科学量化**：将文学描述转化为可比较的数值体系，使跨作品对战具有合理性。不是"谁强谁弱"的主观判断，而是基于原作事实的客观量化。

3. **跨作品对战**：通过"次元压制"机制解决跨作品战力不平衡问题，既保留原作设定的真实性（神话角色确实强于凡人），又通过分级匹配保证游戏可玩性。

## 二、故事与世界设定

### 2.1 万卷浮生宇宙世界观

在时空的尽头，存在一座超越维度的图书馆——**万卷阁**。万卷阁收藏着所有公版小说世界的原本，每一卷书都是一个独立运转的次元世界。书中的角色在万卷阁中以"灵体"形态存在，保留着原作中的全部能力与记忆。

万卷阁由远古的**守护者一族**世代守护。守护者拥有"唤灵"之力，能够召唤书中角色为己所用，在万卷阁的"演武场"中进行跨作品对决。这种对决不仅是战斗，更是对不同文学作品力量体系的验证与致敬。

### 2.2 跨作品融合设定

#### 2.2.1 时空裂缝

万卷阁中存在无数**时空裂缝**，连接着不同的小说世界。当守护者召唤角色时，角色通过时空裂缝从原作世界来到演武场。这解释了为什么不同时代、不同作品的角色可以同场竞技：

- 孙悟空（唐代小说《西游记》）可以与迪迦（现代特摄《奥特曼》）对战
- 项羽（汉代史书《史记》）可以与诸葛亮（元末小说《三国演义》）交锋
- 后羿（上古神话《山海经》）可以与林黛玉（清代小说《红楼梦》）相遇

#### 2.2.2 次元差距的叙事解释

不同作品的力量体系存在"次元差距"，这是世界观的内在规则：

- **神话级世界**（西游记、封神演义、山海经、白蛇传、奥特曼）：角色拥有超自然神力，可呼风唤雨、移山填海、变化无穷。这些世界的"法则"允许超越物理极限的力量存在。
- **凡人级世界**（三国演义、水浒传、红楼梦、史记）：角色虽有非凡武勇或智谋，但受限于凡人躯体与物理法则。

当神话级角色对战凡人级角色时，"次元压制"自然生效——这不是游戏平衡的妥协，而是世界观的客观规律。正如孙悟空的金箍棒一万三千五百斤砸向凡人，凡人无论如何勇武也难以承受。

### 2.3 时空设定

#### 2.3.1 角色如何共存

万卷阁的"演武场"是一个**超然空间**，独立于所有小说世界的时间流。在这里：

- 时间是静止的，角色不会衰老
- 角色保留原作中的全部记忆与能力
- 角色的"灵体"形态可以承受战斗损伤，但不会真正死亡
- 战斗结束后，角色回到原作世界，记忆清零

#### 2.3.2 战力量化的合理性解释

万卷阁的守护者一族开发了**"阅卷术"**——一种通过阅读原作文本，科学量化角色能力的秘术。阅卷术将角色的描述转化为7个维度的数值：

- **力量（power）**：物理攻击力，如项羽"霸王举鼎"→power: 96
- **速度（speed）**：敏捷与先手，如孙悟空"筋斗云十万八千里"→speed: 99
- **智力（intelligence）**：谋略与判断，如诸葛亮"智多近妖"→intelligence: 98
- **防御（defense）**：抗击打能力，如孙悟空"铜皮铁骨"→defense: 95
- **特殊能力（special_ability）**：法术与神通，如姜子牙"封神术"→special_ability: 98
- **生命值（hp）**：生命力，神话角色通常远高于凡人
- **法力值（mp）**：法术能量，法系角色通常较高

### 2.4 核心矛盾

### 2.4.1 世界失衡

故事的主线矛盾源于**古籍世界的失衡**。万卷阁中某些强大的邪灵角色（如封神演义中的妖魔、山海经中的凶兽）开始突破原作世界的边界，侵蚀其他小说世界。这导致：

- 《三国演义》的世界出现妖魔乱入
- 《红楼梦》的大观园被异象侵扰
- 各小说世界的剧情开始错乱

### 2.4.2 守护者的使命

玩家作为万卷阁新任守护者，使命是：

1. **收集角色**：通过阅读（上传）小说文本，召唤更多角色灵体
2. **恢复秩序**：击败侵蚀其他世界的邪灵，恢复各小说世界的剧情
3. **演武验证**：在演武场进行跨作品对战，验证角色实力，提升守护者等级
4. **终极对决**：最终面对导致世界失衡的幕后黑手

### 2.5 故事主线

玩家作为"万卷阁守护者"的旅程分为五个章节：

| 章节 | 名称 | 核心剧情 | 解锁角色 |
|------|------|----------|----------|
| 第一章 | 万卷初启 | 玩家继承守护者之位，学习唤灵术，召唤首批角色 | 孙悟空、诸葛亮、关羽 |
| 第二章 | 三国风云 | 三国世界被妖魔侵蚀，玩家协助刘关张平乱 | 鲁智深、武松、貂蝉 |
| 第三章 | 封神之战 | 封神世界邪灵暴走，玩家卷入封神大战 | 姜子牙、哪吒、二郎神 |
| 第四章 | 神话降临 | 上古神话苏醒，玩家面对山海经凶兽 | 后羿、白素贞 |
| 第五章 | 万卷归一 | 幕后黑手现身，玩家集结所有角色进行终极对决 | 迪迦、项羽、林黛玉 |

### 2.6 世界观规则

### 2.6.1 角色能力来源

所有角色能力必须符合以下规则：

1. **原作依据**：每个技能、每个属性数值必须有原作文本支撑，记录在`source_basis`字段
2. **形态变化**：角色可拥有多个形态（如孙悟空"齐天大圣""斗战胜佛"），每个形态的加成需符合原作设定
3. **技能类型**：技能分为physical_attack（物理）、magic_attack（法术）、speed_attack（速度）、transform（变化）、heal（治疗）、summon（召唤）、passive（被动）七类

### 2.6.2 PVP对战的叙事包装

PVP对战在叙事中被称为**"演武"**——守护者之间的友谊切磋。演武场设有"次元结界"，保证战斗不会真正伤害角色灵体。战斗叙述由万卷阁的"史官"（AI）实时记录，形成文学化的战斗报告。

### 2.6.3 为什么孙悟空可以和迪迦对战

这是世界观的核心设定问题，解释如下：

1. **万卷阁超越时空**：万卷阁收藏所有公版/开放IP作品，包括现代特摄（奥特曼属公版角色形象）。孙悟空和迪迦都是"书中角色"的灵体形态。
2. **演武场超然空间**：演武场独立于原作世界，所有角色在此以灵体形态存在，能力完整保留。
3. **次元压制规则**：两者同属"神话级"（西游记为神话小说，奥特曼为光之巨人），故正常对战，无次元压制。
4. **战力数值化**：通过"阅卷术"，两者的能力被量化为可比数值，使对战具有客观性。

### 2.6.4 为什么角色有战力数值

战力数值是"阅卷术"的产物，叙事合理性体现在：

1. **守护者一族的秘术**：阅卷术是守护者世代传承的能力，可将文学描述转化为数值
2. **原作事实驱动**：数值不是随意设定，而是基于原作中的具体描述（如"金箍棒一万三千五百斤"对应power: 98）
3. **科学量化体系**：7维度数值+分级系统，使不同作品的角色具有可比性
4. **演武验证**：数值的准确性通过演武场的实际对战验证，形成闭环

## 三、角色设计

### 3.1 角色设计原则

万卷浮生的角色设计遵循四大原则：

1. **原作还原**：角色属性、技能、形态必须忠实于原作，每个数值都有文本依据
2. **平衡性**：同级角色战力接近，跨级差距保留但可控
3. **多样性**：覆盖物理、法术、速度、辅助等多种定位
4. **可识别性**：角色技能名称、叙述文案具有强烈的原作辨识度

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | number | 角色唯一ID（1-14） |
| name | string | 角色名称 |
| grade | string | 分级（S/A/B/C） |
| source | string | 出处作品 |

| skills | array | 技能列表（5个） |
| forms | array | 形态列表（基础+进阶） |
| source_basis | string | 原作依据文本 |

**技能类型枚举**：

| type | 说明 | 伤害基数公式 |
|------|------|-------------|
| physical_attack | 物理攻击 | power × 1.2 |
| magic_attack | 法术攻击 | special_ability × 1.2 |
| speed_attack | 速度攻击 | speed × 1.2 |
| summon | 召唤攻击 | power × 0.6 |
| transform | 变化技能 | 无伤害（施加buff/debuff） |
| heal | 治疗技能 | 恢复maxHP×20% |
| passive | 被动技能 | 不主动释放 |

## 四、关卡设计

### 4.1 剧情养成关卡

#### 4.1.1 有向节点图结构

剧情关卡采用**有向节点图（DAG）**结构，每个角色的剧情由多个节点组成，节点之间存在有向边表示剧情推进顺序。

```json
{
  "character_id": 1,
  "story_graph": {
    "nodes": [
      {
        "id": "node_1",
        "type": "story",
        "title": "石猴出世",
        "content": "花果山上一仙石迸裂，化出一只石猴...",
        "next": ["node_2"]
      },
      {
        "id": "node_2",
        "type": "choice",
        "title": "拜师学艺",
        "content": "石猴决定渡海求学，你选择...",
        "choices": [
          { "text": "拜菩提祖师为师", "next": "node_3a" },
          { "text": "自立为王", "next": "node_3b" }
        ]
      },
      {
        "id": "node_3a",
        "type": "skill_unlock",
        "skill": "七十二变",
        "next": ["node_4"]
      },
      {
        "id": "node_4",
        "type": "battle",
        "enemy": "混世魔王",
        "next": ["node_5"]
      },
      {
        "id": "node_5",
        "type": "form_unlock",
        "form": "齐天大圣",
        "next": ["node_6"]
      },
      {
        "id": "node_6",
        "type": "ending",
        "title": "齐天大圣",
        "content": "孙悟空自封齐天大圣，剧情完结"
      }
    ]
  }
}
```

#### 4.1.2 节点类型

| 节点类型 | type | 功能 | 示例 |
|----------|------|------|------|
| 剧情节点 | story | 展示剧情文本，推进故事 | "石猴出世" |
| 选择节点 | choice | 提供分支选择，影响后续剧情 | "拜师学艺" |
| 技能解锁 | skill_unlock | 解锁角色技能 | 解锁"七十二变" |
| 形态解锁 | form_unlock | 解锁角色形态 | 解锁"齐天大圣" |
| 战斗节点 | battle | 触发与NPC的战斗 | 对战"混世魔王" |
| 结局节点 | ending | 剧情章节完结 | "齐天大圣"结局 |

#### 4.1.3 选择分支设计

选择分支影响角色成长方向：

- **技能分支**：不同选择解锁不同技能（如拜师学艺解锁七十二变，自立为王解锁其他技能）
- **形态分支**：选择影响可解锁的形态（如选择"取经"解锁斗战胜佛，选择"称王"解锁齐天大圣）
- **属性加成**：部分选择提供永久属性加成（如选择"修炼"获得intelligence+5）

### 4.2 PVP擂台设计

#### 4.2.1 匹配机制

基于实际`ws-server.js`的`matchQueue`实现：

| 匹配方式 | 触发条件 | 流程 |
|----------|----------|------|
| 快速匹配 | 玩家点击"匹配对战" | 加入matchQueue → 真人优先匹配 → 10秒超时匹配机器人 |
| 邀请对战 | 玩家选择在线玩家邀请 | 发送invite → 对方接受/拒绝 → 开始战斗 |
| 机器人对战 | 玩家选择机器人 | 直接开始战斗 |

**匹配流程详解**：

1. 玩家选择角色后点击匹配，加入`matchQueue`队列
2. `tryMatchmaking()`检查队列，若有2名真人则立即匹配
3. 若等待超过10秒，自动匹配空闲机器人
4. 匹配成功后发送`match_found`消息，延迟1秒开始战斗

#### 4.2.2 战斗流程

基于`startBattle`函数实现：

```
匹配成功 → 发送battle_start → 延迟1秒 → 执行executeBattle（完整计算）
→ 每1.5秒发送一个battle_round → 战斗结束发送battle_end → 保存战绩
```

#### 4.2.3 胜负判定

| 判定条件 | 说明 |
|----------|------|
| HP归零 | 一方currentHp ≤ 0，另一方获胜 |
| 回合上限 | 12回合未分胜负，按HP百分比判定 |
| HP百分比相同 | 按speed判定先手方获胜 |

#### 4.2.4 奖励机制

| 奖励类型 | 获取条件 | 内容 |
|----------|----------|------|
| 战绩 | 每场战斗 | wins+1或losses+1 |
| 经验 | 每场战斗 | 胜利+50exp，失败+20exp |
| 排行榜排名 | 累计胜场 | 按wins降序排列 |
| 成就 | 特殊条件 | 首胜、十连胜、秒杀王等 |

### 4.3 关卡难度曲线

| 章节 | 推荐等级 | 敌人分级 | 难度 | 解锁内容 |

#### 5.2.4 数值公式

**伤害计算公式**（基于`calculateDamage`函数）：

```
base = 技能类型基数
  physical_attack: power × 1.2
  magic_attack: special_ability × 1.2
  speed_attack: speed × 1.2
  summon: power × 0.6
  其他: power × 1.0

base += special_ability × 0.3  // 特殊能力加成

skillMult = skill.multiplier  // 技能倍率
critRate = min(0.35, (speed + intelligence) / 400) + (crit_up buff ? 0.25 : 0)
isCrit = random() < critRate
critMult = isCrit ? 1.5 : 1.0

def = defense × (defense_up buff ? 1.5 : 1)
defReduction = 1 - (def / (def + 200))

attackDownMult = attack_down buff ? 0.7 : 1.0
gradeMult = getGradeMult(attacker, defender)  // 等级压制
dimensionMult = getDimensionMult(attacker, defender)  // 次元压制

damage = base × skillMult × critMult × defReduction × attackDownMult × gradeMult × dimensionMult

// 次元秒杀：神话打凡人至少80%最大HP
if (dimensionMult >= 3.0):
    damage = max(damage, floor(defender.hp × 0.8))

damage = max(1, round(damage))
isOneShot = damage >= defender.hp × 0.6  // 秒杀判定
```

#### 5.4.4 次元压制机制

基于`MYTH_SOURCES`和`getDimensionMult`实现：

```javascript
const MYTH_SOURCES = ['西游记', '封神演义', '山海经', '白蛇传', '奥特曼'];

function getDimensionMult(attacker, defender) {
    const atkMythic = MYTH_SOURCES.includes(attacker.source);
    const defMythic = MYTH_SOURCES.includes(defender.source);
    if (atkMythic && !defMythic) return 3.0;  // 神话打凡人3倍
    if (!atkMythic && defMythic) return 0.5;  // 凡人打神话0.5倍
    return 1.0;  // 同类正常
}
```

**次元秒杀规则**：当`dimensionMult >= 3.0`时，伤害至少为目标最大HP的80%。

#### 5.4.5 等级压制机制

基于`getGradeMult`函数实现：

```javascript
const GRADE_VALUE = { 'S': 4, 'A': 3, 'B': 2, 'C': 1 };

function getGradeMult(attacker, defender) {
    const diff = GRADE_VALUE[attacker.grade] - GRADE_VALUE[defender.grade];
    if (diff >= 2) return 2.0;   // S打B：2倍
    if (diff === 1) return 1.5;  // S打A / A打B：1.5倍
    if (diff <= -2) return 0.5;  // B打S：减半
    if (diff === -1) return 0.8; // A打S / B打A：8折
    return 1.0;                  // 同级正常
}
```

**架构特点**：
- Express与WebSocket共享同一HTTP端口（PORT 3000）
- 战斗引擎为纯计算模块，无外部API调用
- SQLite启用WAL模式提升并发读写性能
- 静态文件由Express直接托管

### 7.3 数据库Schema

基于实际`db.js`实现，启用WAL模式和外键约束：

#### 7.3.1 users表（玩家账号）

```sql
CREATE TABLE IF NOT EXISTS users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT UNIQUE NOT NULL,
    password TEXT NOT NULL,          -- bcrypt哈希
    nickname TEXT,
    avatar TEXT DEFAULT '🎮',
    wins INTEGER DEFAULT 0,
    losses INTEGER DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

#### 7.3.2 characters表（系统预置角色）

```sql
CREATE TABLE IF NOT EXISTS characters (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    grade TEXT NOT NULL,             -- S/A/B/C
    source TEXT NOT NULL,            -- 出处作品
    gradient TEXT,                   -- 渐变CSS
    image TEXT,                      -- SVG路径
    stats_json TEXT NOT NULL,        -- 7维度属性JSON
    skills_json TEXT NOT NULL,       -- 技能列表JSON
    forms_json TEXT NOT NULL,        -- 形态列表JSON
    source_basis TEXT                -- 原作依据
);
```

#### 7.3.3 user_characters表（玩家角色库）

```sql
CREATE TABLE IF NOT EXISTS user_characters (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    character_id INTEGER NOT NULL,
    level INTEGER DEFAULT 1,
    exp INTEGER DEFAULT 0,
    unlocked_forms TEXT DEFAULT '[]',  -- 已解锁形态JSON
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (character_id) REFERENCES characters(id),
    UNIQUE(user_id, character_id)
);
```

#### 7.3.4 battles表（对战记录）

```sql
CREATE TABLE IF NOT EXISTS battles (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    player1_id INTEGER NOT NULL,
    player2_id INTEGER NOT NULL,
    player1_char_id INTEGER NOT NULL,
    player2_char_id INTEGER NOT NULL,
    winner_id INTEGER,
    total_rounds INTEGER,
    battle_log_json TEXT,            -- 完整回合日志JSON
    narration TEXT,                  -- AI战斗叙述
    battle_type TEXT DEFAULT 'pvp',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (player1_id) REFERENCES users(id),
    FOREIGN KEY (player2_id) REFERENCES users(id)
);
```

#### 7.3.5 数据库配置

```javascript
db.pragma('journal_mode = WAL');    // WAL模式提升并发
db.pragma('foreign_keys = ON');     // 启用外键约束
```
