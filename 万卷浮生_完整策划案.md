## 一、产品概述

### 1.1 产品名称

**万卷浮生**

### 1.2 一句话简介（30字以内）

> 把任意公版小说角色变成可对战卡牌，AI科学量化战力，1v1自动战斗PVP见分晓。

### 1.3 核心价值主张

万卷浮生是一个 Web 端 AI 角色卡牌 PVP 游戏。玩家上传公版小说文本，AI 自动提取角色并按统一维度科学计算战力属性，生成可对战角色卡牌；玩家通过剧情养成角色，再与其他玩家的角色进行 1v1 自动战斗 PVP，用战斗结果回答"孙悟空 vs 迪迦谁更厉害"这类跨作品战力讨论。

**核心价值三要素：**

1. **AI 造卡** —— 任意公版小说 → 几分钟生成可对战角色卡牌，内容供给近乎无限
2. **科学量化** —— 统一 7 维度 + 原作还原算法，跨作品角色科学可比
3. **真打一场** —— 1v1 自动战斗 PVP，把"嘴上吵"变成"用结果说话"

### 1.4 差异化定位

| 维度 | 传统卡牌游戏（阴阳师/Fate/GO/崩坏星穹铁道） | 万卷浮生 |
| --- | --- | --- |
| 角色来源 | 官方预设，开发者策划 | 玩家上传任意公版小说，AI 提取 |
| 角色数量 | 数十至数百，受产能限制 | 近乎无限，公版小说库 + 用户原创 |
| 战力依据 | 官方数值策划设定 | AI 根据原作描写科学计算，有据可查 |
| 养成方式 | 抽卡 + 升级 + 装备 | 推进原作剧情，选择影响成长方向 |
| PVP 模式 | 同作品内角色对战 | 跨作品梦幻对决（孙悟空 vs 迪迦） |
| 创作门槛 | 玩家无法创造角色 | 任何玩家几分钟"造"出一批卡牌 |

### 1.5 古籍活化公益价值（附加赛道）

万卷浮生参加**古籍活化**附加赛道，公益价值体现在三方面：

1. **经典文学的新生触达** ——《红楼梦》《西游记》《三国演义》《聊斋》《封神演义》等古籍经典以"收集对战"方式触达新一代读者。玩家为养强孙悟空，会主动去读《西游记》了解技能来源，让古籍从"课本必读"变成"游戏必查"。
2. **古籍角色数字化建档** —— AI 提取过程本身即是对古籍角色的一次结构化数字化：姓名、身份、性格、关系网、关键经历、能力描写均以 JSON 结构存储，形成可检索的古籍角色数据库。
3. **降低古籍阅读门槛** —— 通过"养成 → 查原著 → 再养成"的循环，引导玩家主动阅读古籍原文。MVP 版本预置四大名著角色库，上线后逐步扩展至唐宋传奇、明清小说、外国公版名著。

## 二、市场分析

### 2.1 行业背景

**AI + 游戏趋势：**
- 2024-2026 年 AIGC 游戏方向爆发，AI 已能完成长文本理解、角色特征提取、数值平衡、战斗叙述生成等任务，技术条件成熟
- 字节跳动、腾讯、网易均在布局 AI + 游戏赛道，TRAE 大赛主题"世界很大，放手去造"明确鼓励 AI 原生创作
- AI 角色扮演、AI NPC、AI 生成内容（AIGC）成为游戏行业新增长点

**卡牌 PVP 游戏市场：**
- 《阴阳师》运营 8 年仍盈利，验证卡牌 PVP 高粘性商业模式
- 《Fate/GO》《崩坏：星穹铁道》全球年收入超 10 亿美元，市场体量巨大
- 玩家痛点：官方角色产出慢、想要的角色官方不出、跨作品对战无平台

**跨作品战力讨论流量：**
- B 站"孙悟空 vs XXX"类战力对比视频动辄百万播放
- 贴吧"战力吧""跨作品对战吧"活跃用户数十万
- 知乎"XX 和 YY 谁更厉害"问题常年热门，证明用户需求强烈

### 2.2 竞品对比表

| 对比维度 | 阴阳师 | Fate/GO | 崩坏：星穹铁道 | **万卷浮生** |
| --- | --- | --- | --- | --- |
| 角色来源 | 日本阴阳师 IP | Fate 系列 IP | 米哈游原创 | **任意公版小说 + 用户原创** |
| 角色数量 | ~200 | ~300 | ~50 | **近乎无限（AI 提取）** |
| 养成方式 | 抽卡 + 升级 + 御魂 | 抽卡 + 升级 + 礼装 | 抽卡 + 升级 + 遗器 | **剧情推进 + 选择分支** |
| PVP 模式 | 同作品内斗技 | 基本无 PVP | 模拟宇宙 PVE | **跨作品 1v1 自动战斗** |
| 战力依据 | 官方数值策划 | 官方数值策划 | 官方数值策划 | **AI 原作还原算法** |
| 创新点 | 日式和风美术 | 剧情向 IP 运营 | 箱庭探索 + 战斗 | **AI 造卡 + 跨作品对战** |
| 玩家创作 | 无 | 无 | 无 | **玩家上传小说造卡** |

### 2.3 市场机会分析

**机会一：AI 技术成熟窗口期**
长文本理解（GLM-4-Flash、Claude、通义千问）+ 结构化输出（JSON Mode）+ 多模型适配层技术均已成熟，AI 提取角色 + 战力量化的成本与准确度达到产品化临界点。

**机会二：跨作品对战蓝海**
现有卡牌游戏均局限于单一 IP 内对战，"孙悟空 vs 迪迦"这类跨作品对战无任何平台满足。万卷浮生是首个用 AI 解决跨作品战力量化与对战的产品。

**机会三：内容供给无限化**
公版小说数量庞大（四大名著、唐宋传奇、明清小说、外国公版名著），每本可提取数十角色，角色卡牌供给近乎无限，彻底打破官方产能瓶颈。

**机会四：天然社交传播**
"孙悟空 vs 迪迦"类话题自带百万级流量，每个对战视频都是免费宣传，获客成本极低，契合短视频时代内容营销。

## 三、目标用户

### 3.1 用户画像

#### 用户一：战力讨论爱好者（核心用户）
- **年龄**：15-35 岁
- **特征**：活跃在 B 站、贴吧、知乎的"XX vs YY"战力对比话题下，渴望"真打一场"见分晓
- **行为**：观看战力对比视频、参与评论区辩论、收藏战力排行榜
- **需求**：科学量化的战力数据 + 可对战平台
- **付费意愿**：中（为本命角色变强付费）

#### 用户二：卡牌对战玩家（次核心用户）
- **年龄**：18-30 岁
- **特征**：喜欢《阴阳师》《Fate/GO》《崩坏：星穹铁道》，但苦于官方角色产出慢、想要的角色官方不出
- **行为**：长期玩 1-2 款卡牌游戏、关注角色平衡性、参与 PVP 排位
- **需求**：更多角色选择 + 公平 PVP 环境
- **付费意愿**：高（卡牌游戏用户付费习惯成熟）

#### 用户三：小说/动漫重度粉丝
- **年龄**：16-28 岁
- **特征**：对某些角色有强烈情感，想"拥有"这个角色并培养他变强
- **行为**：阅读原著、参与同人创作、收藏角色周边
- **需求**：把喜欢的角色变成可养成卡牌
- **付费意愿**：中高（为情感付费）

#### 用户四：直播/短视频内容创作者
- **年龄**：20-35 岁
- **特征**：需要新鲜互动内容，"我用 AI 让孙悟空和迪迦打了一场"自带传播潜力
- **行为**：B 站、抖音、YouTube 内容创作，关注热点话题
- **需求**：可生成对战视频/战斗叙述的内容工具
- **付费意愿**：低（但带来大量免费流量）

### 3.2 使用场景

**场景一：朋友战力争论**
和朋友吵"孙悟空 vs 迪迦谁厉害"吵不出结果 → 各自上传小说养成角色，PVP 见分晓，输的请客。

**场景二：本命角色养成**
喜欢某个小说角色（如《红楼梦》林黛玉），官方游戏里没有 → 自己上传小说，AI 生成角色卡牌，开始剧情养成。

**场景三：直播互动内容**
直播时需要互动内容 → 观众投票选择出战角色，AI 自动战斗，弹幕押注，AI 生成战斗小说式叙述增加观赏性。

**场景四：跨作品梦幻对决**
想看"诸葛亮 vs 梅林""鲁智深 vs 路飞"→ 上传双方原作小说，AI 量化战力，自动战斗生成结果与叙述。

**场景五：古籍角色收集**
为收集强力 S 级角色，主动阅读《西游记》《封神演义》等古籍 → 在游戏中解锁孙悟空、姜子牙等角色卡牌。

### 3.3 痛点与解决方案对照表

| 痛点 | 现状 | 万卷浮生解决方案 |
| --- | --- | --- |
| 跨作品战力讨论无解 | 只能嘴上吵，永远没结论 | AI 量化战力 + PVP 自动战斗，用结果说话 |
| 卡牌游戏角色受限 | 官方出什么玩家玩什么，想要的角色官方不出 | 玩家上传任意公版小说，AI 提取角色生成卡牌 |
| 小说角色无法"拥有" | 读完小说只能记住角色，无法"养成"他 | 把角色变成可养成卡牌，升级、学技能、解锁形态 |
| 战力对比无科学依据 | 各作品设定不同，无法直接对比 | AI 用统一 7 维度量化所有角色，数值有原作依据 |
| 古籍阅读门槛高 | 课本必读，年轻读者无兴趣 | 游戏化收集对战，引导主动阅读原著 |
| 直播内容同质化 | 缺乏新鲜互动内容 | AI 跨作品对战 + 战斗叙述，自带话题流量 |

## 四、核心玩法设计

### 4.1 核心循环

```
上传公版小说
      ↓
AI 双轮提取（角色卡 + 战力量化）
      ↓
生成可对战角色卡牌
      ↓
剧情养成（推进节点 → 选择分支 → 成长/解锁技能）
      ↓
PVP 对战匹配（跨作品 1v1）
      ↓
自动战斗（属性计算 + 技能触发 + AI 叙述）
      ↓
获得结果（胜负 + 战斗回放 + 排名变化）
      ↓
（回到养成，继续强化或收集新角色）
```

### 4.2 玩家旅程说明

**第一阶段：造卡（首次体验）**
1. 玩家进入首页，选择"上传小说"或"使用预置角色库"
2. 上传公版小说文本（支持 .txt / .epub / 粘贴文本）
3. AI 在 1-3 分钟内完成双轮提取，生成角色卡牌列表
4. 玩家查看角色卡牌，了解属性、技能、原作依据
5. 选择一个本命角色作为初始角色

**第二阶段：养成（深度体验）**
1. 进入角色剧情有向节点图
2. 推进剧情节点，遇到选择分支
3. 不同选择影响角色成长方向（攻击型/防御型/智慧型）
4. 关键节点解锁新技能或形态变化
5. 重复养成多个角色，构建角色库

**第三阶段：对战（核心价值）**
1. 选择一个角色出战
2. 系统匹配对手（跨作品角色）
3. 自动战斗：双方按速度决定出手顺序，属性计算伤害，技能触发
4. AI 生成战斗小说式叙述
5. 查看战斗回放、属性对比、胜负原因分析
6. 获得排名变化与奖励

**第四阶段：社交传播（裂变）**
1. 分享战斗回放至 B 站/抖音
2. 邀请朋友上传小说对战
3. 参与跨作品对战话题

### 4.3 玩法特色

1. **AI 造卡，内容无限** —— 任意公版小说 → 几分钟生成可对战角色卡牌，打破官方产能瓶颈
2. **科学量化，跨作品可比** —— 统一 7 维度 + 原作还原算法，孙悟空"筋斗云十万八千里"→速度拉满，数值有据可查
3. **剧情养成，选择影响成长** —— 推进原作剧情，选择"大闹天宫"还是"保护唐僧"影响角色成长方向
4. **跨作品梦幻对决** —— 孙悟空 vs 迪迦、诸葛亮 vs 梅林，让永远吵不出结果的战力讨论真正打一场
5. **AI 战斗叙述** —— 根据战斗数据生成小说式叙述，增加观赏性与传播力

## 五、玩法系统

> 本章节基于 `battle-engine.js` 实际实现编写，所有公式与机制均为已落地代码。

### 5.1 角色提取系统

#### 5.1.1 双轮提取流程

公版小说动辄数十万字，采用**两轮提取**保证准确度与可解释性：

```
┌─────────────────────────────────────────┐
│  第一轮 · 角色卡提取（Character Card）   │
│                                         │
│  输入：小说全文（分章节送入 AI）          │
│  AI 任务：逐章扫描，提取每个角色的：     │
│    - 姓名、身份、性格、关系网            │
│    - 关键经历、所有战斗/能力描写         │
│  输出：角色卡 JSON（结构化文本）          │
└─────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│  第二轮 · 战力量化（Power Quantization） │
│                                         │
│  输入：第一轮生成的角色卡                │
│  AI 任务：根据能力描写，按统一 7 维度    │
│          计算战力数值，生成分级与技能    │
│  输出：可对战角色卡牌 JSON（含数值）     │
└─────────────────────────────────────────┘
```

**为什么分两轮？**
- 第一轮专注"理解角色"，避免数值计算干扰角色理解
- 第二轮专注"量化战力"，基于完整角色卡做数值计算，结果更准确
- 玩家可查看"为什么孙悟空力量值是 98"，依据来自第一轮的原作描写引用

```javascript
// battle-engine.js 第 29-39 行
function getGradeMult(attacker, defender) {
    const atkVal = GRADE_VALUE[attacker.grade] || 2;
    const defVal = GRADE_VALUE[defender.grade] || 2;
    const diff = atkVal - defVal;
    if (diff >= 2) return 2.0;   // S打B：2倍
    if (diff === 1) return 1.5;  // S打A 或 A打B：1.5倍
    if (diff <= -2) return 0.5;  // B打S：减半
    if (diff === -1) return 0.8; // A打S 或 B打A：8折
    return 1.0;
}
```

**平衡原则：**
- **同级平衡**：S 级角色间总战力值接近，保证 PVP 公平
- **跨级保留差距**：S 级 vs B 级保留原作差距（2倍伤害）
- **PVP 匹配优先同级**：匹配系统优先匹配同级角色，跨级对战需玩家确认

#### 5.4.3 技能触发逻辑

**AI 选技能逻辑（实际代码）：**

```javascript
function selectSkill(character, currentHp, currentMp) {
    const hpRatio = currentHp / character.stats.hp;
    // 可用技能（排除被动，MP足够）
    const availableSkills = character.skills.filter(s =>
        s.type !== 'passive' && currentMp >= s.mp_cost
    );

    // MP不足时使用普攻
    if (availableSkills.length === 0) {
        return character.skills.find(s => s.mp_cost === 0 && s.type !== 'passive');
    }

    // 血量低于30%优先防御/回血
    if (hpRatio < 0.3) {
        const defensive = availableSkills.find(s => s.type === 'transform');
        if (defensive && Math.random() < 0.5) return defensive;
        const healSkill = availableSkills.find(s => s.type === 'heal');
        if (healSkill && Math.random() < 0.6) return healSkill;
    }

    // MP充足时优先高伤害技能（70%概率选最高倍率）
    const attackSkills = availableSkills.filter(s =>
        ['physical_attack', 'magic_attack', 'speed_attack', 'summon'].includes(s.type)
    );
    attackSkills.sort((a, b) => b.multiplier - a.multiplier);
    if (Math.random() < 0.7) return attackSkills[0];
    return attackSkills[Math.floor(Math.random() * attackSkills.length)];
}
```

| 速度攻击 | speed_attack | 速度×1.2 | 20-60 |

```javascript
        narration += `⚡ 仅${battle.total_rounds}回合，${winner.name}便以碾压之势秒杀${loser.name}！`;
    } else {
        // 正常战斗：按回合叙述
        battle.rounds.forEach(roundData => {
            narration += `${action.actor}施展${action.skill}，${action.narration}。`;
        });
        narration += `经过${battle.total_rounds}回合激战，${winner.name}获胜！`;
    }
}
```

**叙述示例（秒杀局）：**

> 孙悟空施展【如意金箍棒·砸】，金箍棒挟万钧之力砸下，地动山摇。
>
> 这一击挟神话级威能，凡人之躯难以承受。关羽虽勇，终究无法跨越次元的鸿沟。
>
> ⚡ 仅1回合，孙悟空便以碾压之势秒杀关羽！

### 6.3 数据库 Schema

> 基于 `db.js` 第 22-90 行的实际表结构。

#### 6.3.1 玩家账号表 users

```sql
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT UNIQUE NOT NULL,
    password TEXT NOT NULL,           -- bcrypt加密
    nickname TEXT,
    avatar TEXT DEFAULT '🎮',
    wins INTEGER DEFAULT 0,
    losses INTEGER DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

#### 6.3.2 系统预置角色表 characters

```sql
CREATE TABLE characters (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    grade TEXT NOT NULL,              -- S/A/B/C
    source TEXT NOT NULL,             -- 来源小说
    gradient TEXT,                    -- 卡牌渐变色
    image TEXT,                       -- 角色图片路径
    stats_json TEXT NOT NULL,         -- JSON存储属性
    skills_json TEXT NOT NULL,        -- JSON存储技能
    forms_json TEXT NOT NULL,         -- JSON存储形态
    source_basis TEXT                 -- 原作依据
);
```

#### 6.3.3 玩家角色库表 user_characters

```sql
CREATE TABLE user_characters (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    character_id INTEGER NOT NULL,
    level INTEGER DEFAULT 1,
    exp INTEGER DEFAULT 0,
    unlocked_forms TEXT DEFAULT '[]', -- JSON存储解锁形态
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (character_id) REFERENCES characters(id),
    UNIQUE(user_id, character_id)
);
```

#### 6.3.4 对战记录表 battles

```sql
CREATE TABLE battles (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    player1_id INTEGER NOT NULL,
    player2_id INTEGER NOT NULL,
    player1_char_id INTEGER NOT NULL,
    player2_char_id INTEGER NOT NULL,
    winner_id INTEGER,
    total_rounds INTEGER,
    battle_log_json TEXT,             -- JSON存储战斗日志
    narration TEXT,                   -- AI战斗叙述
    battle_type TEXT DEFAULT 'pvp',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (player1_id) REFERENCES users(id),
    FOREIGN KEY (player2_id) REFERENCES users(id)
);
```

## 十、商业模式

### 10.1 盈利路径

#### 路径一：高级 AI 模型订阅（核心收入）

- **免费版**：使用 GLM-4-Flash 免费额度，每日限制 3 次角色提取
- **Pro 版**（¥19/月）：无限角色提取，优先使用高级模型（Claude/GPT-4o），战力量化更精准
- **Creator 版**（¥49/月）：Pro 版权益 + AI 战斗叙述增强 + 战斗回放视频导出 + 自定义角色立绘生成

**价值主张**：高级模型提取的角色更准确、技能更丰富，PVP 更有优势。

#### 路径二：角色卡牌皮肤与立绘（增值收入）

- **免费立绘**：AI 文生图基础立绘
- **付费立绘**（¥6-¥18/个）：高级 AI 文生图（更精美、多风格：国风/二次元/写实）
- **动态皮肤**（¥28/个）：带战斗特效的动态立绘
- **限定皮肤**：节日/活动限定，稀缺性驱动收藏

#### 路径三：战斗回放分享与内容创作（流量变现）

- **免费**：基础战斗回放（文字 + 静态图）
- **付费**（¥3/次或 Pro 版权益）：战斗回放视频导出（带 AI 叙述配音），适合 B 站/抖音创作者
- **创作者激励**：高传播战斗回放给予 Pro 版时长奖励

#### 路径四：PVP 排位赛与赛事（长期收入）

- **免费排位**：基础排位赛
- **付费赛事**（¥10/次报名）：周期性跨作品对战赛事，冠军获限定皮肤 + 奖金
- **企业赞助**：赛事冠名（如"某某杯·万卷浮生跨作品对战联赛"）

### 10.2 成本结构

| 成本项 | 月成本估算（1000 DAU） | 说明 |
| --- | --- | --- |
| AI API 调用 | ¥2,000-5,000 | 角色提取 + 战斗叙述，GLM-4-Flash 免费额度覆盖初期 |
| 服务器（Vercel） | ¥0-200 | Vercel 免费额度足够 MVP，Pro 版 $20/月 |
| 数据库（自建） | ¥100-300 | SQLite 零成本，PostgreSQL 备选 ¥200/月 |
| 文件存储 | ¥50-200 | 小说文本存储 |
| 域名与 CDN | ¥50-100 | 域名 + 静态资源 CDN |
| 内容审核 | ¥500-1,000 | 版权审核 + 违规内容审核（初期人工） |
| **合计** | **¥2,700-6,800/月** | 1000 DAU 阶段 |

**成本优化策略：**
- MVP 阶段充分利用 GLM-4-Flash 免费额度
- 上线版支持玩家自带 API Key（成本转嫁）
- 高配玩家支持本地模型（零成本运行）
- 角色卡牌缓存：相同小说提取结果复用，减少 API 调用

### 10.3 用户增长策略

#### 策略一：社交传播（核心增长引擎）

- **跨作品对战话题**：每个"孙悟空 vs 迪迦"类对战都是天然社交内容
- **战斗回放分享**：一键分享至 B 站/抖音/小红书，带 AI 叙述
- **对战邀请**：玩家邀请朋友上传小说对战，双方各得 1 个月 Pro 版
- **排行榜炫耀**：全服战力排行，激发"我的本命角色最强"攀比

#### 策略二：B 站/抖音内容营销

- **官方账号**：定期发布"跨作品对战"视频（如"诸葛亮 vs 梅林""关羽 vs 亚瑟王"）
- **KOL 合作**：邀请战力讨论类 UP 主体验，"我用 AI 让孙悟空和迪迦打了一场"
- **直播互动**：观众投票选择出战角色，AI 自动战斗，弹幕押注
- **UGC 激励**：玩家创作的优质对战视频给予 Pro 版奖励

#### 策略三：跨作品对战话题运营

- **每周热门对战**：运营团队策划热门跨作品对战（如"三国五虎 vs 漫威复仇者"）
- **节日主题赛事**：中秋节"嫦娥 vs 七仙女"、春节"年兽 vs 饕餮"
- **古籍活化专题**：每月主推一本古籍（《红楼梦》月、《西游记》月），引导玩家阅读原著

#### 策略四：社区运营

- **官方社区**：角色卡牌分享、养成路线讨论、跨作品对战投票
- **角色创作大赛**：玩家上传原创小说提取角色，评选最佳角色卡
- **MOD 支持**（长期）：开放角色立绘、技能特效 MOD，激发 UGC

## 十一、风险评估

### 11.1 技术风险及应对

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| **AI 提取准确度不足** | 角色卡牌质量参差不齐，影响体验 | 1. 双轮提取提升准确度；2. Prompt 持续优化与 A/B 测试；3. 玩家可手动编辑角色卡；4. 社区投票校验 |
| **长文本处理超限** | 数十万字小说超出上下文窗口 | 1. 分章节送入 AI；2. 跨章节角色合并；3. 长文本摘要后再提取；4. 限制单次上传字数（如 50 万字） |
| **战斗平衡性问题** | 某些角色过强/过弱，影响 PVP 公平 | 1. 分级平衡机制（S/A/B/C 同级平衡）；2. 上线后持续数据监控与微调；3. 玩家反馈机制；4. 定期平衡性更新 |
| **AI 模型服务不稳定** | API 不可用导致功能中断 | 1. 多模型适配层，主备切换；2. 本地模型兜底；3. 角色卡牌缓存，已提取角色不依赖实时 API |
| **性能瓶颈** | 高并发下战斗计算慢 | 1. 战斗引擎纯计算，无 API 调用，性能可控；2. Vercel 自动扩容；3. 数据库索引优化 |
| **数值膨胀** | 长期运营后数值失控 | 1. 严格 0-100 分制；2. 总战力值区间限制；3. 定期数值重铸 |

### 11.2 版权风险及应对

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| **非公版小说上传** | 侵犯版权，法律风险 | 1. **仅支持公版小说**（明确告知用户）；2. AI 自动识别疑似非公版内容并拦截；3. 用户举报机制；4. 用户协议明确版权责任 |
| **用户原创小说版权** | 版权归属争议 | 1. 用户上传原创小说需勾选授权协议；2. 角色卡牌版权归用户所有；3. 平台获得展示与对战授权 |
| **角色立绘版权** | AI 文生图可能涉及版权 | 1. 使用合规 AI 文生图服务；2. 不模仿特定画师风格；3. 商用立绘使用付费服务 |
| **版权审核机制** | 违规内容上线 | 1. 预置角色库人工审核；2. 用户上传内容 AI 初审 + 人工复审；3. 违规内容下架机制 |

**公版小说界定标准：**
- 中国作品：作者逝世 50 年以上的作品（如四大名著、《聊斋》《封神演义》）
- 外国作品：符合《伯尔尼公约》公版标准
- 用户原创：用户上传自己拥有版权的作品
- 不支持：仍在版权期内的商业作品（如《斗破苍穹》《三体》等）

### 11.3 运营风险及应对

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| **用户留存低** | DAU 下降，商业模式不可持续 | 1. PVP 高粘性玩法（验证过的卡牌 PVP 模式）；2. 持续新增角色库；3. 赛季制排位；4. 社区运营 |
| **内容质量参差** | 用户上传低质量小说影响体验 | 1. 预置高质量角色库（四大名著）；2. 角色卡牌评分机制；3. 优质内容推荐；4. 低质量内容过滤 |
| **社区管理** | UGC 内容违规 | 1. 内容审核机制（AI + 人工）；2. 社区公约；3. 举报与封禁机制；4. 敏感词过滤 |
| **获客成本上升** | 增长放缓 | 1. 社交传播降低获客成本；2. 跨作品对战话题自带流量；3. KOL 合作；4. 古籍活化公益传播 |
| **竞品模仿** | 同类产品出现 | 1. 快速迭代建立先发优势；2. 角色库积累形成壁垒；3. 社区生态护城河；4. 持续创新玩法 |

### 11.4 风险应对总原则

1. **技术风险**：多模型适配 + 缓存策略 + 持续优化
2. **版权风险**：严格公版限制 + AI 审核 + 用户协议
3. **运营风险**：PVP 高粘性 + 社交传播 + 社区运营
4. **整体原则**：MVP 快速验证 → 数据驱动迭代 → 长期运营积累
